# MyPurse 开发合规收尾检查

## 何时触发

以下任一条件满足时调用本 Skill：
1. 用户说"收尾""合规检查""检查文档""提交一下"等
2. 用户确认代码改动验收通过（"OK""没问题""就这样"）
3. Agent 完成一批代码改动后自行判断该收尾了

## 核心流程

按以下 9 条逐条检查执行，不可跳步：

### 1. 编译验证

```bash
./gradlew assembleDebug
```

失败则先修代码，不继续后续步骤。

### 2. 更新开发日志

在 `docs/3-开发过程/开发日志.md` 末尾追加本次会话记录（参考 `references/log-template.md`），然后同步到 `docs/3-开发过程/开发日志-精简版.md`。

### 3. 更新问题记录（如有 Bug 修复）

在 `docs/3-开发过程/问题记录.md` 末尾追加，使用统一模板（现象 → 根因 → 修复 → 教训 → 涉及文件），然后同步到 `docs/3-开发过程/问题记录-精简版.md`。

### 4. 更新进度追踪（如阶段完成）

打开 `docs/3-开发过程/开发阶段.md`，将本次涉及的子阶段项状态列打 ✅。

### 5. 更新需求文档（如需求变更）

如果交互行为或设计决策有实质性变化：更新 `docs/2-需求规格/requirements.md` 正文 + 第 10 章变更记录。

### 6. 删除临时文件

清理：`build_output.txt`、`compile_errors.txt`、以及其他调试/排查产生的临时文件。

### 7. Git 提交

```bash
git add -A
git commit -m "type(scope): 简短描述"
```

### 8. 告知 commit hash

在回复中明确告知 commit hash 和简要内容。

### 9. 告知文档改动

列出本次修改了哪些文档文件（6 类项目文件 + 3 份规范文件），无改动也要明确说"本次无文档改动"。

## 注意事项

- 第 1 条（编译）必须在第 2-6 条之前完成
- 第 7-9 条是 Git 收尾，与 2-6 条同等重要
- 完整规范细节见 `references/compliance-checklist.md`
