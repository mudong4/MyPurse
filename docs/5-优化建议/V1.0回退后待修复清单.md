## 一、立即修复（本次会话完成）

### 1.1 `CategoryDefDao.kt` SQL NULL 比较语法错误 🔴

- **问题**：`getCategoryByName` 和 `getCategoryByNameAndSign` 使用 `parent_id IS :parentId`
- **根因**：SQL 中 `IS` 只能用于 `IS NULL`/`IS NOT NULL`，不能与参数绑定混用。当 `parentId` 非 null 时，SQL 变成 `parent_id IS 123`（非法语法），Room 编译可能通过但运行时出错
- **修复**：改为 `(parent_id = :parentId OR (parent_id IS NULL AND :parentId IS NULL))`
- **文件**：`data/local/dao/CategoryDefDao.kt`
- **评估**：真实 Bug，必须修复。这是 trae 会话 #24 中唯一一个无争议的修复项

### 1.2 `BudgetViewModel.kt` 架构违规 — 直接注入 DAO 🔴

- **问题**：`BudgetViewModel` 直接注入 `BudgetDao`，绕过了 Repository 抽象层
- **根因**：`TransactionRepository` 接口已定义 `getBudget()` 和 `observeBudget()` 方法，但 BudgetViewModel 没有使用
- **影响**：违反 MVVM + Clean Architecture 的依赖倒置原则（ui → domain ← data），使 ViewModel 直接依赖数据层
- **修复**：改为注入 `TransactionRepository`，通过 Repository 方法访问预算数据
- **文件**：`ui/budget/BudgetViewModel.kt`
- **评估**：架构一致性修复。当前项目虽小，但架构不一致会给后续维护者制造困惑

### 1.3 `TransactionListViewModel.kt` 硬编码 `Int.MAX_VALUE` 🟡

- **问题**：分类/年/月/周视图中使用 `Int.MAX_VALUE` 作为 `limit` 参数
- **位置**：`loadCategoryView()`、`loadYearView()`、`loadMonthView()`、`loadWeekView()` 共 4 处
- **影响**：语义不清。虽然 Room Flow 查询不会真的拉取全量，但 `Int.MAX_VALUE` 作为 SQL LIMIT 值是不合理的
- **修复**：定义 `MAX_LOAD_SIZE = 500` 常量替代
- **文件**：`ui/transactionlist/TransactionListViewModel.kt`
- **评估**：防御性改进，语义清晰化

## 二、V1.1 时机合适再做

### 2.1 UseCase 拆分 — ViewModel 减负

- **当前状态**：`TransactionListViewModel.kt` 694 行，臃肿但可控
- **何时做**：V1.1 新增功能前，先拆后加。避免在臃肿的 ViewModel 上叠加新逻辑

- **何时做**：当需要增加第三种流水类型时，常量化变得必要

## 三、暂不做（过度设计）

### 3.1 Repository 拆分为 4 个子接口

- **评估**：当前 166 行单一接口用注释分组，维护成本低。拆分需要额外 Hilt 绑定，调用方需注入多个接口
- **触发条件**：接口膨胀到 200+ 行，且出现只依赖单一子集的调用方
- **延迟成本**：低。方法签名不变，只是把注入类型从 1 个变成 N 个

### 3.2 TimeNavigation 公共组件

- **评估**：流水列表和统计页的时间导航差异大（粒度、位置、快捷跳转方式均不同），强行抽取导致参数膨胀
- **触发条件**：出现第三个需要时间导航的页面时（经典"三场景"原则）
- **延迟成本**：极低。两个页面完全独立，改一个不影响另一个

### 3.3 ColorConfig 统一颜色配置

- **评估**：当前 Color.kt + AppColors.kt + Theme.kt 三层管理已能工作。新增 object 包裹只是改变调用方式
- **触发条件**：当用户需要自定义颜色（设置页新增配色方案选择），或颜色硬编码超过 5 处
- **延迟成本**：低
