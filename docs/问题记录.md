### 问题 #1：分类管理拖拽排序性能卡顿

- **发现日期**：2026-06-15
- **严重程度**：🟡 体验问题（PC 端已可感知，移动端预期更严重）
- **所属模块**：分类管理页 (E)

**现象**：
拖拽分类条目移动时，画面有明显卡顿，不够跟手。

**根因分析**：
1. `onDrag` 回调每帧（~60fps）都会调用 `onMoveCategory`，触发 LazyColumn 重组和逐条数据库写入
2. `animateDpAsState` 为每个列表项创建动画状态对象，增加重组开销
3. 每次拖拽过程中逐条执行 `updateSortOrder`，产生大量数据库 I/O

**解决方案**：
1. `onDrag` 中仅累积 `dragOffsetY`，不做任何数据变更；实际的位置交换逻辑移至 `onDragEnd` 执行
2. 将 `animateDpAsState` 替换为简单的条件判断（`if (isDragging) 8.dp else 0.dp`）
3. 新增 `batchUpdateSortOrder` DAO 方法，使用 `@Transaction` 在单次事务中批量更新所有受影响分类的排序序号

**涉及文件**：
- `CategoryManageScreen.kt`：拖拽手势逻辑重构
- `CategoryManageViewModel.kt`：`onMoveCategory` 改为调用批量更新
- `CategoryDefDao.kt`：新增 `batchUpdateSortOrder`
- `CategoryRepository.kt` / `CategoryRepositoryImpl.kt`：新增批量更新接口及实现

### 问题 #2：拖拽排序后点击位置错乱

- **发现日期**：2026-06-15
- **严重程度**：🟠 功能缺陷
- **所属模块**：分类管理页 (E)

**现象**：
拖拽"餐饮"到新位置后，再次点击"餐饮"，响应的是原来位置（拖拽前的位置）的分类。

**根因分析**：
1. **闭包捕获了旧值**：`LazyColumn` 中 `itemsIndexed` 的 `pointerInput(category.id)` 的 key 未因重排序而改变，导致手势检测回调中捕获的 `index` 和 `categories` 仍是拖拽前的旧值
2. **UI 状态被覆盖**：`loadCategories()` 中的 `collect` 使用了 `_uiState.value.copy(...)` 而非 `current.copy(...)`，导致 `expandedParentId` 等 UI 状态字段在数据刷新时被重置为默认值

**解决方案**：
1. 在 `itemsIndexed` 的 lambda 中使用 `rememberUpdatedState` 包装 `index` 和 `categories`，确保长生命周期的 gesture 回调始终读取最新值
2. 将 `_uiState.value.copy(...)` 改为 `current.copy(...)`，保留所有 UI 状态字段

**涉及文件**：
- `CategoryManageScreen.kt`：添加 `rememberUpdatedState`，修复手势回调中的闭包捕获
- `CategoryManageViewModel.kt`：修复 `loadCategories()` 中的状态合并方式

**教训**：
- Compose 中 `pointerInput` 的 key 不变时，内部 lambda 捕获的变量不会自动更新，必须用 `rememberUpdatedState` 桥接
- `StateFlow.collect` 中使用 `value.copy()` 会丢弃 collect 之前的并发更新，应使用 `current.copy()`

### 问题 #3："恢复默认"按钮无响应

- **发现日期**：2026-06-15
- **严重程度**：🟠 功能缺陷
- **所属模块**：分类管理页 (E)

**现象**：
1. 点击"恢复默认"按钮后，分类顺序未恢复，UI 无变化
2. 按钮仅恢复支出侧还是同时恢复收入侧不明确

**根因分析**：
1. `restoreDefaultCategories` 方法只处理了"添加缺失的一级分类"，完全忽略了：
   - 已存在分类的 `sortOrder` 恢复为默认值
   - 子分类的恢复
2. 子分类数据（`CategoryDefaults.subCategories`）从未被使用

**解决方案**：
重写 `restoreDefaultCategories` 方法，执行以下三步：
1. 将已存在的一级分类的 `sortOrder` 重置为默认值
2. 添加缺失的一级分类（含正确的 `sortOrder`）
3. 遍历 `CategoryDefaults.subCategories`，添加缺失的子分类

**涉及文件**：
- `CategoryRepositoryImpl.kt`：重写 `restoreDefaultCategories`

**待确认**：
- 用户提问"恢复默认是把收入和支出都恢复吗"——当前实现仅恢复当前选中的那一侧（支出或收入），通过顶部 Tab 切换后需分别点击恢复。后续可考虑增加"恢复全部"选项。

### 问题 #4：分类模式下 LazyColumn 重复 Key 导致闪退

- **发现日期**：2026-06-16
- **严重程度**：🔴 阻塞
- **所属模块**：流水列表页 (D)

**现象**：
在流水列表底部工具栏，将分组方式从"时间"切换到"一级分类"或"二级分类"后，App 立即闪退。

**根因分析**：
`GroupedTransactionList` 中，LazyColumn 的 item key 始终使用 `group.rangeStart.toString()`。在分类模式下，所有 `TimeGroup` 的 `rangeStart` 都是同一个时间范围起始时间戳（如 `1718409600000`），导致多个 group header 产生相同的 key `"header_1718409600000"`，LazyColumn 检测到重复 key 后抛出异常崩溃。

**解决方案**：
根据分组模式区分 key 生成策略：
- 时间模式（`GroupMode.TIME`）：继续使用 `rangeStart.toString()`，因为每个时间段的起始时间天然唯一
- 分类模式（`CATEGORY_L1` / `CATEGORY_L2`）：改用 `group.label`（如"餐饮"、"交通"），因为分类名天然唯一

修改位置：`TransactionListScreen.kt` 的 `GroupedTransactionList` 函数，将 key 生成逻辑改为：
```kotlin
val groupKey = if (groupMode != GroupMode.TIME) group.label else group.rangeStart.toString()
```

**涉及文件**：
- `TransactionListScreen.kt`：修复 `GroupedTransactionList` 中的 `groupKey` 生成逻辑

**教训**：
- 当分组依据不是时间时，不能假设 `rangeStart` 唯一
- LazyColumn 的 `item(key = ...)` 的 key 必须在整个列表范围内唯一，重复 key 会导致崩溃而非警告

### 问题 #5："StandaloneCoroutine was cancelled" 系统 Toast

- **发现日期**：2026-06-16
- **严重程度**：🟡 体验问题（仅 debug 构建出现，release 不受影响）
- **所属模块**：全局 / 流水列表页 (D)

**现象**：
操作过程中偶尔弹出 Toast 提示 "StandaloneCoroutine was cancelled"，不影响功能但观感不佳。

**根因分析**：
Android 14+ 系统在 debug 构建中会检测到 `CancellationException` 并通过 Toast 展示。本项目 `TransactionListViewModel` 中 `loadData()` 和 `searchData()` 使用 `loadJob?.cancel()` 取消上一个加载协程，当用户快速切换筛选条件时触发 cancel，系统将其作为"未捕获异常"弹出 Toast。

**解决方案**：
1. `TransactionListViewModel` 中：`cancel()` 前先注册 `invokeOnCompletion` 空回调吸收异常；协程内部显式 catch `CancellationException` 并重新抛出（遵循协程规范），与普通异常分开处理
2. `MyPurseApplication` 中：设置全局 `UncaughtExceptionHandler`，对 `CancellationException` 静默处理（仅打 log），其他异常仍走原有崩溃流程

**涉及文件**：
- `TransactionListViewModel.kt`：`loadData()`、`searchData()` 的 cancel 和异常处理逻辑
- `MyPurseApplication.kt`：新增全局未捕获异常处理器

**补充说明**：
正常的 release 应用不会出现此提示，因为该检测是 Android 系统仅在 debuggable=true 的构建中启用的调试行为。但即便如此，处理方案本身（全局异常处理器 + 区分 CancellationException）也是 Android 开发的最佳实践。
