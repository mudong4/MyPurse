## 0. 适用范围与快速开始

### 0.1 适用范围

本指南包含两类内容，用标签区分：

| 标签 | 含义 | 说明 |
|------|------|------|
| `[通用]` | 与技术栈和项目类型无关，任何项目都应遵守 | 规划方法论、反模式、验收标准 |

Agent 使用本指南时：
- `[通用]` 内容视为强制规则

当需要根据需求文档规划开发阶段时，按以下步骤执行：

```
1. 通读需求文档，提取所有功能模块
2. 阅读第 1 章"规划方法概览"，判断项目适合哪种规划方式

5. 按第 4 章"输出模板"格式化输出
```

## 1. 规划方法概览 [通用]

开发阶段规划没有唯一正确的方式，选择取决于项目特征。以下是业界常见的 4 种方式：

### 1.1 能力驱动的顺序阶段规划

```
需求 → 功能按依赖/复杂度/优先级拆解 → 按能力分组为阶段 → 严格顺序执行
```

**特征**：阶段之间严格线性依赖，阶段 N 完成后才能开始阶段 N+1。按能力（数据层、页面组、辅助功能）而非按页面分组。

**适用场景**：
- 需求相对明确，不太需要用户反馈来调整方向
- 数据层高度共享（多页面依赖同一套 Entity/DAO/Repository）
- 单人或小团队开发，无需并行

**优势**：简单易执行，进度可预测，每个阶段产出清晰

**局限**：
- 问题发现晚——质量验收往往攒到最后一个阶段集中做
- 不支持部分交付——全部阶段做完才有完整产品
- 不适应需求变化——中途加需求时阶段规划需要大幅调整

**典型阶段结构**（以有数据库的应用为例）：

| 阶段 | 内容 |
|------|------|
| 0 | 项目初始化（构建系统、DI、数据库、导航壳、主题） |
| 1 | 数据层基础（Entity、DAO、Repository、Domain Model） |
| 2-N | 按能力分组的页面/功能组 |
| N+1 | 收尾（质量验收、性能、边界，拆为子阶段而非杂项大杂烩） |

### 1.2 迭代/增量式规划

```
需求 → 选定核心子集 → 完整实现（UI→DB）→ 交付 → 追加下一子集 → 循环
```

**特征**：每次迭代交付一个可用的产品子集（从 UI 到 DB 的完整切片），逐步扩展功能。

**适用场景**：
- 需求不完全明确，需要用户反馈来验证方向
- 希望尽早交付可用版本
- 功能模块间耦合度低

**优势**：早期交付价值，风险早暴露，用户反馈驱动调整

**局限**：
- 需要良好的架构解耦，否则后期重构成本高
- 数据层可能需要多次扩展（每次迭代加表/字段），Schema 变更管理更复杂
- 总工作量可能比顺序规划更大（重复设计/重构）

**典型阶段结构**：

| 迭代 | 交付目标 |
|------|---------|
| 迭代 1 | 最小可用（如：只能记账+查看列表） |
| 迭代 2 | 追加统计能力 |
| 迭代 3 | 追加预算/管理能力 |
| ... | 逐步扩展 |

### 1.3 MVP + Sprint 规划

```
需求 → 定义 MVP（最小可行产品）→ 拆为若干 Sprint → 每个 Sprint 1-2 周 → 快速交付
```

**特征**：先定义最小可行产品，再以短期冲刺（Sprint）迭代。每个 Sprint 结束时有可演示的产出。

**适用场景**：
- 创业产品、需要快速验证市场
- 团队采用敏捷开发流程
- 产品方向可能随时调整

**优势**：极快交付核心价值，节奏感强，适应变化

**局限**：
- 规划粒度粗，技术方案可能不够深思熟虑
- 容易累积技术债——每个 Sprint 都在赶交付，架构优化被推迟
- 需要有一个能做产品决策的 Product Owner

### 1.4 功能垂直切片规划

```
需求 → 按功能切片 → 每条切片从 UI 到 DB 完整实现 → 切片间尽量独立
```

**特征**：每条切片是从用户视角的完整功能链（前端→API→数据库），切片间独立开发、独立部署。

**适用场景**：
- 微服务架构、前后端分离项目
- 团队可按切片并行开发
- 功能间数据耦合度低

**优势**：每条切片独立可交付，团队可并行，故障隔离

**局限**：
- 共享数据层（如公共用户表）需要特殊处理，可能重复实现
- 切片间的数据一致性需要额外设计
- 对小型单仓库项目增加不必要的复杂度

### 1.5 如何选择

| 判断维度 | → 选顺序阶段 | → 选迭代/增量 | → 选 MVP+Sprint | → 选垂直切片 |
|---------|-------------|-------------|----------------|-------------|
| 需求明确度 | 明确 | 不完全明确 | 可能随时变 | 按功能可拆 |
| 数据层耦合 | 高（共享 Entity/DAO） | 中 | 低 | 低 |
| 团队规模 | 1-2 人 | 2-5 人 | 3-8 人 | 可并行分配 |
| 是否需要部分交付 | 不需要 | 需要 | 急需 | 按功能交付 |
| 项目复杂度 | 中小型 | 中型 | 任意 | 中大型 |

**默认推荐**：如果无法判断，选择"能力驱动的顺序阶段规划"——它最简单、最可预测，适合大多数 AI 辅助开发的个人/小团队项目。后续章节的工作流和输出模板也以这种方式为基础编写，但标注了哪些步骤在其他方式下需要调整。

## 2. 工作流 [通用]

### Step 1：读取需求

通读需求文档。如项目有 `docs/` 目录，还应阅读：
- `1-项目概览/项目介绍.md` — 项目架构和关键决策
- `4-规范与指南/通用开发规范.md` — 通用开发标准（如有）
- `4-规范与指南/开发规范.md` — 项目特定编码规范（如有）

识别：
- 所有功能模块及其描述
- 模块间依赖关系（如模块 B 需要模块 A 的数据层）
- 非功能需求（性能、安全、UX）
- 平台/技术约束

### Step 2：功能拆解

从需求文档中提取每个功能单元，确定：

| 属性 | 判断方式 |
|------|----------|
| 依赖 | 什么必须先存在才能构建这个功能？（数据层？另一个功能？） |
| 复杂度 | 低（简单 UI 或单个 DAO）、中（完整页面+ViewModel）、高（多页面+复杂逻辑） |
| 优先级 | P0（核心价值主张）、P1（重要但不阻塞）、P2（锦上添花） |

### Step 3：阶段分组

应用以下规则（标注了适用范围，非当前规划方式的规则标注了适配提示）：

1. **基础设施先行** `[通用]`：阶段 0 始终是项目初始化（构建系统、DI 框架、导航壳、主题等）。有数据库的项目还包括数据库初始化
2. **数据层在 UI 之前** `[项目经验]`：有持久层的项目，阶段 1 应先完成 Entity、DAO、Repository、Domain Model，再开始 UI。无数据库的项目跳过此规则
3. **按能力分组，不按页面分组** `[项目经验]`：适用于数据层高度耦合的项目。一个阶段 = 一个可独立验证的能力。包含"首页+记一笔+列表"的阶段太粗糙。若采用迭代/增量方式，改为"按交付子集分组"
4. **最后一个阶段禁止是杂项大杂烩** `[通用]`："打磨+性能+测试+边界"塞一起是反模式。拆为 2-3 个子阶段，各有序号和验收标准
5. **每个阶段包含测试** `[通用]`：原型期之后，每个阶段必须包含核心逻辑的单元测试作为子任务。具体测试对象（DAO/UseCase/Service）取决于技术栈
6. **数据库 Schema 变更必须有 Migration 子任务** `[项目经验]`：有 Room/SQLite 等本地数据库的项目适用。永远不要假设 `fallbackToDestructiveMigration()` 是安全的

### Step 4：验证阶段规划

输出前检查：

- [ ] 规划方式已声明，选择理由合理（见第 1 章）
- [ ] 每个需求章节至少被一个阶段覆盖
- [ ] 依赖关系正确（没有阶段依赖后面的阶段）
- [ ] 没有"杂项"阶段 — 每个阶段有清晰主题
- [ ] 每个阶段有具体、可验证的验收标准
- [ ] 测试策略显式声明（原型 vs 生产）
- [ ] 规划考虑了项目成熟度（原型 → 功能开发 → 维护）
- [ ] `[项目经验]` 规则已根据当前项目特征判断适用性

### Step 5：输出

按下方"输出模板"格式化输出。

## 4. 经验教训（反模式清单）

> 以下反模式按适用范围标注。`[通用]` 反模式与项目类型无关，`[项目经验]` 反模式来源于数据库驱动的 Android 应用（MyPurse V1.0），其他项目类型需判断是否适用。

### 反模式 1：阶段按页面拆分而非按能力拆分 [项目经验]

> **适用前提**：数据层高度耦合的项目（多页面共享同一套 DAO/Repository）。数据层独立或按功能解耦的项目，按页面/功能切片分组反而是合理的。

**表现**：阶段 2=首页+记一笔，阶段 3=流水列表，阶段 4=统计页……

**后果**：
- 阶段间耦合：阶段 3 做完发现 DAO 查询不够用，回头改阶段 2 的数据层
- 改动一个 DAO 影响多个页面，但没有回归测试

**正确做法**：先拆分数据层能力（Entity/DAO/Repository），再拆分页面组。阶段 B 依赖阶段 A 的**数据层**，不依赖阶段 A 的 UI。

### 反模式 2：最后一个阶段是杂项大杂烩 [通用]

**表现**：最后一阶段 = 空状态 + 错误处理 + 性能 + UX + 边界 + 测试，全塞在一起。

**后果**：无法判断进度、没有优先级排序。

**正确做法**：拆成 2-3 个子阶段，各有序号和验收标准。例如：
- Na：空状态 + 错误处理（用户直接感知，性价比最高）
- Nb：性能验证（插入大量数据 + 测试帧率）
- Nc：边界测试（极端金额、跨年数据）

### 反模式 3：测试策略未纳入阶段规划 [通用]

**表现**：全程零测试，阶段规划中没有"测试"的位置。

**后果**："修 A 坏 B"反复发生。进入维护期后改动任何代码都是"盲改"。

**正确做法**：
- 每个阶段完成后必须写核心路径的单元测试（DAO、UseCase 等）
- 如果项目处于原型验证期可以不强制，但阶段规划中必须**显式声明**这一点
- 一旦进入维护期，修改任何模块前必须确保有测试覆盖

### 反模式 4：数据库 Schema 变更风险被低估 [项目经验]

> **适用前提**：使用本地数据库（Room/SQLite/CoreData 等）的项目。无本地数据库或使用云数据库的项目不适用。

**表现**：开发期贪方便用 `fallbackToDestructiveMigration()`，忘了有真实测试数据，版本升级后数据全部丢失。

**正确做法**：阶段规划中，每次涉及 Schema 变更的阶段，必须包含 Migration 编写作为子任务。

### 反模式 5：未区分原型期和维护期的阶段粒度 [通用]

**正确做法**：

| 阶段 | 粒度 | 测试 | 文档 |
|------|------|------|------|
| 原型验证期（V0.x） | 粗（一次做多个页面） | 不强制 | 需求文档即可 |
| 功能开发期（V1.0） | 中（一次做一个页面组） | 核心模块有测试 | 完整文档体系 |
| 维护迭代期（V1.1+） | 细（一次做一个功能点） | 全量回归测试 | 同步更新所有文档 |

### 反模式 6：阶段验收标准模糊 [通用]

**表现**：阶段描述是"流水列表"或"统计页"，没有明确的"做完的标准"。

**正确做法**：每个阶段必须有可验证的验收标准。例如：
- ❌ "首页 UI 完成"
- ✅ "首页能显示当前结余（正/负颜色）、今日/本周/本月/今年汇总、预算进度条"

## 5. 关键规则（强制）

- **禁止创建"阶段 N：杂项"阶段。** `[通用]` 拆为编号子阶段。
- **验收标准必须可验证。** `[通用]` 不是"UI 完成"，而是具体可检查的行为描述。
- **区分原型与生产阶段。** `[通用]` 原型阶段可粗、可跳测试；生产阶段必须细、必须测试。
- **数据库 Schema 变更从来不是小事。** `[项目经验]` 有本地数据库的项目，必须在相关阶段中包含 Migration 作为显式子任务。
- **选择规划方式时必须声明理由。** `[通用]` 输出阶段规划时，第一行必须写明采用的规划方式（见第 1 章）及选择理由。
- **自我更新遵循通用开发规范第 14 章。** `[通用]` 本文档的经验教训如需沉淀为规范条目，按《通用开发规范》14.2 沉淀标准判断去向，14.3 更新流程执行。

## 6. 本指南的使用风险与边界 [通用]

### 风险 1：工作流以"能力驱动的顺序阶段"为默认

**默认假设**：本指南工作流（第 2 章 Step 3）和输出模板（第 3 章）以"能力驱动的顺序阶段规划"为基础编写。

**不适用场景**：采用迭代/增量、MVP+Sprint、垂直切片等其他规划方式的项目。

**缓解**：Agent 应先按第 1 章选择规划方式，再调整工作流和模板的具体内容。选择非默认方式时，必须在输出中声明并说明调整点。

### 风险 2：经验教训的来源局限

**事实**：反模式 1（按页面拆分）和反模式 4（Schema 变更）来自 MyPurse V1.0（单人开发的 Android 记账 App，数据层高度耦合）。其他类型项目可能不适用。

**缓解**：标注了 `[项目经验]` 的反模式，Agent 需根据当前项目特征判断是否适用，不可机械套用。

### 风险 3：机械套用测试要求

**本指南要求**：原型期之后每个阶段必须写测试。

**可能的问题**：对于"快速验证想法"的 hobby 项目或一次性工具，强制测试会拖慢节奏。

**缓解**：Agent 必须在输出阶段规划时**显式声明**该项目当前处于原型期还是维护期，并据此调整测试策略。如果用户明确说"先不写测试"，应尊重。

### 风险 4：输出模板可能过度猜测

**模板要求**：每个阶段预估"新增文件"和"修改文件"列表。

**可能的问题**：对于大型项目或 Agent 不熟悉的技术栈，预估文件列表可能严重不准，给用户错误预期。

**缓解**：文件清单标注"预估"，且允许用户反馈后调整。Agent 不应在不确定时强行填满，可写"待阶段开始时确定"。
