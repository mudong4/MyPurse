# 通用软件开发规范（Agent 可执行版）

> **文档版本**：V3.8 (2026-06-20)

> **目标读者**：AI Agent（优先）与人类开发者  
> **设计原则**：每一条都是可执行指令，不含模糊描述。结构化表达，段落尽量少。

---

## 0. 适用范围与文档关系 (V3.0)

### 0.1 适用范围

本规范包含两类章节，用标签区分：

| 标签 | 含义 | 适用项目 |
|------|------|---------|
| `[通用]` | 与技术栈无关，任何项目都应遵守 | 所有项目 |
| `[Android]` | Android/Kotlin 生态特有 | Android 应用 |
| `[数据库]` | 有持久化层的项目 | 使用数据库的项目 |

**Agent 接手项目时**：先读 0.2 节了解文档关系，再按 0.3 节顺序阅读项目文档。对于标注了不适用标签的章节，跳过即可。

### 0.2 文档关系

```
通用开发规范（本文档）← 代码级规范（写代码时实时约束）
     ↕ 互相引用
项目启动指南          ← 流程引导（项目生命周期第①②阶段：需求澄清 + 文档初始化）
     ↓ 衔接
开发阶段规划指南       ← 流程引导（项目生命周期第③阶段：拆分开发阶段）
```

三份文档定位不同：
- **通用开发规范**：你**写代码时**对照的约束清单
- **项目启动指南**：你**接到新项目时**的引导流程
- **开发阶段规划指南**：你**规划怎么分阶段开发时**的引导流程

三份文档的详细内容各自独立，但版本号管理统一遵循本文档 6.5 节。

### 0.3 Agent 初始阅读顺序与结束动作 (V3.8)

当 Agent 接手一个项目时，按以下角色顺序阅读项目文档（文件名和存放位置因项目而异，按内容角色查找即可）：

```
1. 项目概览文档       → 项目全貌、架构、关键技术决策（如 项目介绍.md）
2. 开发过程日志       → 最近做了什么、改了哪些文件（如 开发日志.md）
3. 问题/Bug 记录      → 已知 Bug 和坑点，避免重复踩坑（如 问题记录.md）
4. 需求规格文档       → 功能规格和交互设计
5. 项目编码规范       → 该项目特定的编码约束（如 开发规范.md）
6. 开发进度追踪       → 当前进度、待完成事项（如 开发阶段.md）
```

**Agent 完成任务后必须执行以下步骤（不可跳过）**：

> **执行顺序**：先做代码自检（第 12 章），再执行收尾步骤。自检不通过则先修代码，收尾只记录已通过自检的代码。

> **⚠️ 文档记录时机（V3.7 新增）**：文档更新（第 1-4 步）应在**用户确认验收通过后**执行，而非每次代码修改后。原因：
> - 同一问题可能修改多轮才成功，中间轮次的"解决方案"未必是最终方案
> - 用户验收通过是一个明确的触发点，不会被遗忘
> - 避免产生大量冗余记录
>
> 但以下情况可以**中间记录**：
> - 根因分析已完成且确定，即使代码还在改（根因不会变）
> - 教训已经明确，值得立即沉淀（避免遗忘）

1. 更新开发日志文档（追加本次改动记录）
2. 如有 Bug 修复 → 更新问题记录文档
3. 如阶段完成 → 更新进度追踪文档
4. 如需求变更 → 更新需求文档（正文直接替换 + 在独立的变更记录章节追加记录，含原文摘要和决策理由）
5. 删除所有临时文件（编译输出、调试日志等）
6. **主动告知用户本次修改了哪些文档**（见下方"文档变更通知规范"）

#### 文档变更通知规范 (V3.4)

**强制要求**：Agent 每次会话结束时，必须在回复中**明确告知用户**本次修改了哪些文档文件。

**为什么需要**：
- 文档改动不像代码改动那样容易在 `git status` 中一眼识别（文档分散在不同目录）
- 用户无法直接感知 Agent 是否执行了收尾步骤中的文档更新
- 缺少通知会导致用户对文档状态无感知，积累"文档到底有没有改"的信任问题

**通知格式**（简洁列出即可，不需表格）：

```
## 本次文档改动

### N-类文件
- `路径/文件名`：改动摘要
  （无改动则写"无"）

### 规范文件
- `路径/文件名`：改动摘要（如有，需特别标注）
  （无改动则写"无"）
```

**分类说明**：
- **N 类文件**：指 6.1 节定义的 6 类项目文件（项目概览/开发日志/问题记录/需求规格/进度追踪/项目编码规范）
- **规范文件**：指 0.2 节定义的 3 份规范文档（通用开发规范/项目启动指南/开发阶段规划指南），规范文件的修改需**特别标注**因为影响面跨项目

**铁律**：每次会话结束必须执行此通知。即使"本次无文档改动"也要明确说出来，不可省略。

#### 0.3 收尾快速检查表（Agent 每次任务完成时逐条过）

> **为什么需要这张表**：收尾动作分散在 0.3 节（文档更新+通知）和 6.0 节（git commit+告知 hash）。Agent 在长对话中容易遗漏某一步。合并为一张 Checklist 后逐条对照，不会漏。

```
☐ 1. 代码自检（第 12 章）
☐ 2. 更新开发日志文档
☐ 3. 更新问题记录文档（如有 Bug 修复/教训）
☐ 4. 更新进度追踪文档（如阶段完成）
☐ 5. 更新需求文档（如需求变更）
☐ 6. 删除所有临时文件（编译输出、调试日志等）
☐ 7. git commit（按 6.0 节格式：type(scope): 描述）
☐ 8. 告知用户 commit hash + 简要内容（6.0 节第 5 条）
☐ 9. 告知用户本次修改了哪些文档（上方"文档变更通知规范"）
```

**使用方式**：任务完成后，从第 1 条开始逐条确认，不可跳步。第 7-9 条是 Git 收尾，与第 1-6 条同等重要。

## 1. 命名规范 [通用] (V1.0)

### 1.1 通用规则（Agent 执行时逐条对照）

| # | 规则 | 检查方式 |
|---|------|----------|
| 1 | 名称表达意图，不表达实现 | 读名字能猜到它是干什么的？ |
| 2 | 同一概念全项目用同一个词 | grep 确认没有 `getXxx` / `fetchXxx` 混用 |
| 3 | 布尔变量以 `is`/`has`/`should`/`can` 开头 | `isLoading`, `hasError`, `canSubmit` |
| 4 | 列表变量用复数名词 | `users`, `transactions` |
| 5 | 禁止单字母变量（循环变量 `i`, `j`, `k` 除外） | — |
| 6 | 禁止匈牙利前缀（`strName`）、成员前缀（`m_count`） | — |

### 1.2 文件/目录命名

| 场景 | 规则 | 示例 |
|------|------|------|
| 目录 | 按功能命名，不按技术角色 | `user-auth/` 而非 `controllers/` |
| 测试目录 | 与源码目录结构镜像 | `src/test/.../user-auth/` |
| 多词文件名 | 用语言惯例（连字符或驼峰） | `user-auth.ts` 或 `UserAuth.ts` |

## 2. 架构规范（Agent 强制执行）[通用] (V1.0)

### 2.1 分层依赖方向

```
[表现层]  →  [业务层]  ←  [数据层]
  UI/API      UseCase     Repository实现
              ↑ 接口在此定义 ↑
```

**铁律**：
- 表现层只调用业务层，不直接访问数据库/网络
- 业务层定义接口，数据层实现接口
- 业务层不依赖任何框架（Android SDK、Express、Spring 等）

### 2.2 类设计原则（写代码时逐条自检）

| 原则 | 执行指令 | 自检问题 |
|------|----------|----------|
| SRP 单一职责 | 一个类/文件只做一件事 | 能用一句话说清这个类的职责吗？ |
| OCP 开闭原则 | 加新功能时通过新增类实现，不修改已有类 | 如果加一种新类型，要改几个 switch/if？ |
| DIP 依赖倒置 | 构造函数注入接口，不 new 具体类 | 这个类能换一个数据库实现而不改代码吗？ |
| ISP 接口隔离 | 接口不超过 5 个方法 | 实现类有没有空方法？ |

### 2.3 模块拆分规则

- 两个类如果总是一起修改，放在同一个模块
- 两个模块如果存在 A→B 的依赖，禁止再出现 B→A
- 禁止按技术角色拆分顶层目录（❌ `controllers/` + `services/` + `models/`）

## 3. 代码质量（Agent 写代码时强制执行）[通用] (V1.0)

### 3.1 函数约束

| 约束 | 阈值 | 超过时的处理 |
|------|------|-------------|
| 行数上限 | ≤ 30 行 | 拆分为 2+ 个私有函数 |
| 参数上限 | ≤ 4 个 | 封装为参数对象（data class/struct） |
| 嵌套层级 | ≤ 3 层 | 提前 return / 抽取子函数 |
| 抽象层级 | 一个函数内不跳层 | 不要既有 SQL 拼接又有业务判断 |

### 3.2 注释规则

| 规则 | 执行指令 |
|------|----------|
| 写"为什么" | 注释解释动机和约束，不重复代码内容 |
| 禁止废话 | `// i 加 1` 对 `i += 1` 是无用注释，删除 |
| 公开 API 必注释 | 所有 public 函数/类必须有：用途、参数说明、返回值说明 |
| TODO 格式 | `// TODO(author): 待做什么 — 预计何时可移除` |

### 3.3 错误处理规则

| 规则 | 执行指令 |
|------|----------|
| 禁止吞异常 | `catch {}` 为空必须加注释说明原因，否则视为 Bug |
| 错误信息可操作 | 消息告诉用户/调用方"发生了什么 + 可以怎么处理" |
| 异常转换 | 数据层异常 → 转换为业务层错误类型 → 再传给表现层 |
| 可预期失败用 Result | 用 `Result<T, Error>` 或 `Optional` 类型，不用异常 |
| 不可预期错误抛异常 | 如磁盘满、内存不足 |

### 3.4 禁止写法清单

| 禁止 | 替代方案 |
|------|----------|
| 魔法数字（`if (status == 3)`） | 命名常量（`if (status == STATUS_EXPIRED)`） |
| 巨型类（>300 行） | 按职责拆分为多个类 |
| 复制粘贴代码块 | 抽取为公共函数/组件 |
| 过早优化（没测就优化） | 先写对 → 写测试 → profile → 优化瓶颈 |
| 5 层以上 if-else 嵌套 | 策略模式 / 查表 / `when`/`match` |

## 5. Git 规范 [通用] (V1.0)

### 5.1 提交格式

```
<type>: <简短描述>
```

**type 取值**：`feat` / `fix` / `refactor` / `docs` / `style` / `test` / `chore` / `perf`

**Agent 提交前自检**：
- [ ] 本次提交只做了一件事？
- [ ] 描述用祈使句（"添加" 而非 "添加了"）？
- [ ] 没有混入不相关的文件改动？

### 5.2 分支命名

```
feature/<功能名>     → 新功能
fix/<问题描述>       → Bug 修复
refactor/<模块名>    → 重构
```

### 5.3 版本回退操作 (V3.3)

| 场景 | 命令 | 风险 | 前置检查 |
|------|------|------|----------|
| 回退单个文件到上次提交 | `git checkout -- <file>` | ⚠️ 中高（不可逆，丢弃该文件所有未提交改动） | 确认文件当前状态 `git diff <file>`，确认不需要保留改动 |
| 回退单个文件到上次提交（安全版） | `git stash push -- <file>` | 低（改动保存在 stash 中可恢复） | 无 |
| 撤销最近提交，保留改动 | `git reset --soft HEAD~1` | 低 | 无 |
| 撤销最近提交，丢弃改动 | `git reset --hard HEAD~1` | 🔴 不可逆 | 必须 `git log -1` 确认最新提交内容 |
| 从历史提交开新分支 | `git checkout -b recover-xxx <hash>` | 低 | 无 |

**铁律**：
1. 大改动前先 commit。**每个阶段/会话完成后立即 commit，不攒多个阶段再一起提交**
2. 执行任何可能丢失改动的 git 操作前，必须先用 `git status` 确认当前工作区状态
3. `git checkout -- <file>` 和 `git reset --hard` 的风险等级被修正：前者从"低"改为"中高"（在工作区有大量未提交改动时等同于不可逆删除），后者从"⚠️"改为"🔴"
4. **优先使用 `git stash` 代替 `git checkout --`**：stash 保留改动可恢复，checkout -- 直接丢弃不可逆
5. **AI Agent 特别约束**：Agent 在未提交改动超过 5 个文件时，禁止执行 `git checkout --` 或 `git reset --hard`。应先提示用户提交或 stash

### 5.4 已推送代码的回退 (V3.4)

当错误已经 push 到远程仓库时，回退策略取决于**是否有人基于该提交在开发**：

| 场景 | 命令 | 风险 | 适用条件 |
|------|------|------|---------|
| 撤销已推送的提交（安全版） | `git revert <hash>` | 低（新增一个反向 commit） | **推荐**，适用于所有场景 |
| 撤销已推送的提交（重写历史） | `git reset --hard <hash>` + `git push --force` | 🔴 极高（破坏远程历史） | 仅限：①确认无其他人基于该分支开发；②推送后立即发现错误（几分钟内）；③单人仓库 |
| 回退到某次提交，保留历史 | `git revert HEAD~2..HEAD` | 低 | 需要撤销最近多个 commit |

**铁律**：
1. **push 后默认用 `revert`**，不用 `reset --hard + force push`。revert 不破坏历史，永远可追溯
2. **禁止 force push 到 main/master**（即使仓库设置了分支保护，也应视为不可操作）
3. 如果必须 force push，先在本地备份：`git branch backup-before-force-push`
4. `git revert` 可以 revert 一个 revert（即"撤销撤销"），所以 revert 本身也是可逆的

**为什么 revert 比 reset 安全**：
```
reset --hard:  A → B → C → [X]        # C 彻底消失
revert:        A → B → C → D(反向C)    # C 保留在历史中，D 撤销 C 的改动
```
即使 revert 错了，`git revert D` 就能恢复 C 的改动。而 reset --hard 删除的提交只能靠 reflog 或备份恢复。

### 5.5 灾难恢复 — reflog 兜底 (V3.4)

即使执行了不可逆操作（`reset --hard`、误删分支），Git 的 reflog 在**本地**仍保留 30-90 天的操作记录：

```bash
# 查看所有 HEAD 变动记录
git reflog

# 恢复到某个 reflog 条目（如误 reset 后恢复）
git reset --hard HEAD@{2}   # 恢复到 2 次操作前的状态

# 恢复误删的分支
git checkout -b recovered-branch <hash>
```

**局限**：reflog 仅存在于本地。如果误操作后已经 force push 覆盖了远程，远程的 reflog 不可用（除非使用 GitHub/GitLab 等平台的仓库事件日志）。

**Agent 强制要求**：在执行任何 `reset --hard` 或 `push --force` 之前，必须：
1. 先运行 `git reflog -5` 并记住当前 HEAD 位置
2. 告知用户 reflog 恢复方式
3. 创建本地备份分支：`git branch backup-$(date +%Y%m%d-%H%M)`

**回退前安全快照**（推荐操作序列）：
```bash
# 1. 查看当前状态
git status

# 2. 保存快照（二选一）
git stash push -m "backup before risky operation"   # 保存到 stash
# 或
git diff > ../backup_$(date +%Y%m%d_%H%M%S).patch   # 保存为 patch 文件

# 3. 执行回退操作
# ...

# 4. 如需恢复：git stash pop  或  git apply ../backup_*.patch
```

## 6. 开发过程管理（Agent 强制遵守）[通用] (V2.0)

### 6.0 阶段完成提交规则 (V3.3)

> **为什么需要这条规则**：AI Agent 阶段式开发中，多个独立会话的改动共享同一个工作区。如果中间没有提交 checkpoint，一次 git 误操作就会丢失跨会话的累积成果。详见 MyPurse 项目问题 #18。

**强制要求**：
1. **每个阶段/会话完成后，Agent 必须立即 commit**，不攒多个阶段再一起提交
2. **commit 前确认**：`git status` 检查改动范围，确保只包含本阶段的文件
3. **commit message 格式**：`<type>(<scope>): <简短描述>`，如 `feat(settings): 设置页完整实现 + CSV导入导出`
4. **例外**：如果改动完全在临时/实验性分支上且确认不会合并，可以不提交
5. **提交后必须告知用户**：commit 完成后，在回复中明确告知 commit hash 和简要内容。push 后同样告知。不可静默提交

> ⚠️ **收尾联动**：commit 只是收尾的一部分。提交后必须继续执行 0.3 节"收尾快速检查表"的第 9 条——告知用户本次修改了哪些文档。不可 commit 后就结束。

**为什么不能等到"全部做完再提交"**：
- 工作区中未提交的改动越多，一次误操作（`checkout --`、`reset --hard`、IDE 意外操作）的破坏面越大
- 阶段式开发中，后续会话的 Agent 依赖前序会话的代码状态。如果前序代码未提交且被误覆盖，后续会话无法恢复
- Git 是唯一的防线——文档记录不能替代代码本身的版本控制

### 6.1 文档体系 (V2.0)

项目应维护以下 **6 类内容**（文件名和存放位置可自定义，但内容角色应齐全）：

| 内容角色 | 应包含的信息 | Agent 更新时机 |
|---------|-------------|---------------|
| 项目概览 | 架构、技术栈、关键技术决策 | 架构变更、新增依赖时 |
| 开发日志 | 每次开发会话的完整改动记录 | **每次会话结束，强制**（完成后须执行 0.3 节第 6 步文档变更通知） |
| 问题记录 | Bug 现象、根因、修复方案、教训 | 发现/修复 Bug 时 |
| 需求规格 | 功能规格、交互设计 | 需求变更时 |
| 进度追踪 | 开发阶段规划、完成状态 | 阶段完成时 |
| 项目编码规范 | 该项目特定的编码约束 | 规范变更时 |

> **说明**：上表描述的是"需要什么内容"，而非"文件必须叫什么"。项目可按自身情况选择文件名（如英文名、不同目录结构），只要 6 类内容的角色都有覆盖即可。Agent 接手项目时，按内容角色定位文档，而非按固定文件名查找。

### 6.1.1 需求变更记录规范 (V2.2)

需求文档维护遵循"**正文干净 + 变更可追溯**"原则：

- **正文**：大功能变更直接替换原文，关键决策处加 HTML 批注说明变更原因，并指向变更记录章节（如 `<!-- 原为X，因Y原因改为Z，详见变更记录章节 -->`）
- **变更记录**：在需求文档末尾设立独立的变更记录章节（章节编号因项目而异），每条记录至少包含 6 列：日期、涉及节、变更类型（新增/删除/简化/修正/推迟）、变更内容、原文摘要（修改前关键内容）、决策理由（为什么这样改）
- **重大变更**（功能删除/重构）：在变更记录章节中附上被替换的原文全文，格式：`> **原章节全文**（Vx.x 版本）：`

目标：正文保持最新版本的可读性，想了解历史决策看变更记录章节。

### 6.1.2 文档间逻辑关系 (V3.1)

6 类文件不是孤立的，它们在开发过程中存在明确的数据流和驱动关系：

```
                        ┌─────────────────────────────────────────────────────┐
                        │              三份规范文档（元层）                    │
                        │  ┌───────────────┐ ┌───────────────┐ ┌────────────┐│
                        │  │ 项目启动指南   │ │ 开发阶段规划  │ │ 通用开发   ││
                        │  │ 驱动①②阶段    │ │ 指南驱动③阶段 │ │ 规范全程   ││
                        │  └──────┬────────┘ └──────┬────────┘ └─────┬──────┘│
                        │         │                 │                │        │
                        └─────────┼─────────────────┼────────────────┼────────┘
                                  │ 产出            │ 产出           │ 约束
                                  ▼                 ▼                ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                        6 类项目文件（实例层）                                │
│                                                                             │
│  ①需求规格 ──────→ ③进度追踪 ──────→ ⑥项目编码规范                        │
│  (做什么)          (分几步做)          (怎么做)                              │
│      │                 │                  │                                 │
│      │                 │ 驱动             │ 约束                             │
│      ▼                 ▼                  ▼                                 │
│  ②项目概览 ←──── ④开发日志 ←──── ⑤问题记录                               │
│  (全局视图)       (做了什么)        (踩了什么坑)                            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**各角色的定位**：

| 文件角色 | 核心问题 | 谁产出/驱动 | 谁消费 |
|---------|---------|------------|--------|
| 需求规格 | 做什么？ | 项目启动指南（①②阶段） | 开发阶段规划指南（拆分阶段的输入）、开发过程（验证功能完整性） |
| 项目概览 | 项目长什么样？ | 项目启动指南（初始化骨架）→ 开发过程（持续补充架构决策） | Agent 接手项目时的第一个阅读对象（0.3 节第 1 项） |
| 进度追踪 | 分几步做？当前到哪了？ | 开发阶段规划指南（③阶段拆分）→ 开发过程（标记完成状态） | Agent 每次会话开始时确认当前进度、决定下一步 |
| 开发日志 | 这次会话做了什么？ | Agent 每次会话结束强制写入 | 后续会话的上下文恢复、问题回溯 |
| 问题记录 | 踩了什么坑？怎么修的？ | Agent/人类 发现和修复 Bug 时写入 | 规范自我更新的输入源（第 14 章）、后续开发避免重蹈覆辙 |
| 项目编码规范 | 这个项目怎么写代码？ | 项目启动指南（初始化）→ 从通用开发规范中筛选适用条目 | 开发过程中实时约束代码风格和模式 |

**关键数据流**：

1. **需求 → 进度**：需求规格的功能列表是开发阶段规划的输入。进度追踪中的每个阶段必须能追溯到需求中的功能点
2. **进度 → 日志**：进度追踪决定"这一步做什么"，开发日志记录"这一步实际做了什么"。阶段完成时，进度追踪的状态更新以开发日志的内容为依据
3. **日志 → 问题**：开发日志中记录的 Bug 和异常，提取到问题记录中独立管理（问题记录需要根因分析和举一反三的教训，日志只需记录现象）
4. **问题 → 规范**：问题记录中的教训，按第 14 章沉淀标准判断是否提升为规范条目
5. **规范 → 编码规范**：通用开发规范是跨项目通用约束，项目编码规范从中筛选适用条目并补充项目特有约束
6. **概览 ← 多方**：项目概览是"汇总层"——架构变更时更新、新增依赖时更新，它自身不产生新信息，但保证后来者能快速建立全局认知

### 6.4 临时文件清理 (V3.7)

Agent 在当前任务完成、用户验收通过后必须删除以下文件：
- `compile_result.txt` / `compile_errors.txt` 等编译输出
- `debug_log.txt` 等调试日志
- 其他为排查问题临时生成的文件

**触发时机**：与文档记录一致，在用户验收通过后执行，而非"会话结束前"（会话可能被新任务打断，此时临时文件可能还需要保留用于继续排查）。

### 6.5 版本号管理规范 (V2.5)

> **背景**：项目中存在两类版本号，格式和含义不同，需明确区分以避免混淆。

#### 6.5.1 两类版本号定义

| 类型 | 含义 | 格式 | 示例 | 适用场景 |
|------|------|------|------|---------|
| **文档修订版本** | 该文档自身第几次修订 | `V主.次` | `V2.3` | 所有项目文档（规范、日志、需求等） |
| **产品功能版本** | 产品功能发布批次 | `V主.次.补丁` | `V1.2.0` | 需求文档功能标签、App 关于页 |

两者**相互独立**——文档修订版本 V2.3 ≠ 产品功能版本 V2.3。

#### 6.5.2 文档修订版本递增规则

| 变化类型 | 版本位变化 | 示例 |
|---------|-----------|------|
| 新增/修改/删除正文内容 | 次版本 +1 | `V2.2 → V2.3` |
| 纯格式调整（缩进、标点统一） | 不变 | — |
| 文档结构大改（章节重组、大幅重写） | 主版本 +1，次版本归 0 | `V2.9 → V3.0` |

- **次版本号自然增长，不限位数**：`V2.9 → V2.10` 合法，不需进位
- **递增粒度**：以"会话"或"批次"为单位。同一会话中对同一文档的多次修改，合并为一个版本号
- **修订记录**：一条记录概括该版本的所有改动

#### 6.5.3 产品功能版本递增规则

| 变化类型 | 版本位变化 | 示例 |
|---------|-----------|------|
| Bug 修复、体验优化（不涉及新功能） | 补丁号 +1 | `V1.2.0 → V1.2.1` |
| 新增一个或多个功能模块 | 次版本 +1，补丁归 0 | `V1.2.1 → V1.3.0` |
| 产品定位/架构颠覆性变化 | 主版本 +1，其余归 0 | `V1.3.0 → V2.0.0` |

补丁号可攒一批 Bug 一起升，不需修一个升一次。

#### 6.5.4 版本号标注要求

每个文档建议**两处标注版本号**，且保持同步：

1. **文件头部引用块**（阅读时一眼可见）：
   ```markdown
   > **文档版本**：V2.5 (2026-06-17)
   ```
2. **文末修订记录表格**（追溯变更历史）：
   ```markdown
   | V2.5 | 2026-06-17 | 作者 | 变更内容描述 |
   ```

#### 6.5.5 节级版本标注

建议每个一级节（`##`）标题旁标注该节最后修改时的文档版本号：

```markdown
## 6. 开发过程管理 (V2.0)
### 6.5 版本号管理规范 (V2.3)
```

- 该节内容修改时，将标注更新为当前文档版本号
- 只保留最新版本号，历史追溯在修订记录中完成
- 需求文档中的功能标签（如 `(V1.0)`）是产品功能版本，属不同概念，不随文档修订更新

#### 6.5.6 各文档版本号独立

不同文档的版本号各自独立演进，互不影响。

## 7. 测试规范 [通用] (V1.0)

### 7.1 测试文件位置

```
src/
├── main/.../feature/
└── test/.../feature/    ← 与源码包路径一致
```

### 7.2 测试命名格式

```
<被测方法>_<场景>_<预期行为>
```

示例：`calculateDiscount_expiredCoupon_returnsZero`

### 7.3 必须测试的场景

| 场景类型 | 具体内容 |
|----------|----------|
| 核心业务 | 金额计算、权限校验、状态机转换 |
| 边界值 | null、空字符串、空集合、0、负数、极大值 |
| 异常路径 | 网络超时、数据库失败、文件不存在、输入格式错误 |
| 回归 | 历史上出过 Bug 的代码路径 |

### 7.4 测试数量分布

- 单元测试：占总测试量 ~70%
- 集成测试：~20%
- 端到端测试：~10%

### 7.5 测试在开发流程中的位置

| 开发阶段 | 测试策略 |
|----------|----------|
| 原型验证期（V0.x） | 不强制写测试。目标是快速验证方向，手动测试即可 |
| 功能开发期（V1.0） | 每完成一个模块后立即写核心路径的单元测试（DAO、UseCase） |
| 维护迭代期（V1.1+） | 必须补测试。改代码前先写回归测试，改完后跑全量测试 |

**铁律**：一旦项目进入维护期（有真实用户或数据），修改任何模块前必须确保该模块有测试覆盖。没有测试的修改 = 盲改。

## 8. 数据库变更规范 [数据库] (V2.1)

### 8.1 Schema 变更流程

每次修改数据库 Schema（加表/改字段/加索引），必须执行以下步骤：

```
1. 递增 @Database version
2. 编写对应的 Migration（Room Migration 类）
3. 在 AppDatabase 的 addMigrations() 中注册
4. 验证 Migration 在已有数据上执行不丢失数据
```

### 8.2 fallbackToDestructiveMigration() 使用规则

| 场景 | 是否可用 |
|------|----------|
| version = 1，首次创建 | ✅ |
| 开发期，确定无真实数据 | ✅（但需加 TODO 标记，计划替换） |
| 有任何真实数据 | ❌ 严禁 |

**铁律**：一旦数据库中有真实数据（哪怕只是开发期手动录入的测试数据），必须写 Migration。

### 8.3 数据安全底线

- 数据库变更前，建议先导出 CSV 备份
- 所有 DAO 的 delete/update 操作必须有 where 条件，禁止无条件的 `DELETE FROM table`

## 9. 异步状态管理规范 [Android] (V2.1)

> **跨技术栈原则**：任何使用响应式状态管理的项目都需要区分"加载中/成功/失败"三种状态。本节以 StateFlow 为例，Vue/React 项目应使用对应的响应式方案（如 Pinia/Recoil），但核心原则相同。

### 9.1 StateFlow 初始值规则

StateFlow 必须有初始值。如果初始值不代表"有效状态"，用以下方式之一：

```kotlin
// 方式一：nullable（推荐）
val data: StateFlow<List<Item>?>  // null = 加载中

// 方式二：sealed class（复杂场景推荐）
sealed class UiState {
    data object Loading : UiState()
    data class Success(val data: List<Item>) : UiState()
    data class Error(val message: String) : UiState()
}
```

**禁止**：用看似合理的默认值伪装"已就绪"状态。例如：
```kotlin
// ❌ 禁止
val transactions: StateFlow<List<Transaction>> = MutableStateFlow(emptyList())
// UI 无法区分"正在加载"和"确实没有数据"
```

### 9.2 空列表防护

涉及 `lastIndex`、`coerceIn`、`first()`、`last()` 等操作的异步数据，必须先判空：

```kotlin
// ❌ 危险：异步数据可能为空列表
val idx = selectedIndex.coerceIn(0, items.lastIndex)

// ✅ 安全
if (items.isEmpty()) return
val idx = selectedIndex.coerceIn(0, items.lastIndex)
```

## 10. 阶段规划原则 [通用] (V2.1)

### 10.1 阶段拆分规则

| 规则 | 说明 |
|------|------|
| 按能力拆分，不按页面拆分 | 一个阶段 = 一个可独立验证的能力（如"数据层完整实现"），而非一个页面 |
| 每个阶段有明确的验收标准 | "首页能显示余额"而非"首页 UI 完成" |
| 最后一个阶段不放杂项 | "收尾优化"必须拆成 2-3 个子阶段，各有序号 |
| 阶段间减少耦合 | 阶段 B 依赖阶段 A 的数据层，不依赖阶段 A 的 UI |

### 10.2 原型期 vs 维护期

| | 原型验证期（V0.x） | 功能开发期（V1.0） | 维护迭代期（V1.1+） |
|------|------|------|------|
| 阶段粒度 | 粗（一次做多个页面） | 中（一次做一个页面组） | 细（一次做一个功能点） |
| 测试要求 | 不强制 | 核心模块有测试 | 全量回归测试 |
| 文档要求 | 需求文档即可 | 完整文档体系 | 同步更新所有文档 |

### 10.3 阶段结束的 Checklist (V3.7)

每完成一个阶段（用户验收通过后），必须确认：
- [ ] 功能按需求规格正常工作（手动验证）
- [ ] 没有引入新的编译警告
- [ ] 开发日志已追加本次改动记录
- [ ] 如有 Bug 修复 → 问题记录已更新
- [ ] 如有需求变更 → 需求文档已更新
- [ ] 进度追踪文档状态已标记为完成

> **记录时机**：与 0.3 节一致，Checklist 中的文档更新项在用户验收通过后执行，非代码提交后立即执行。

## 11. 性能与安全（Agent 写代码时逐条检查）[通用] (V2.1)

### 11.1 性能

| 规则 | 执行指令 |
|------|----------|
| 先测后优 | 没有 profile 数据不优化 |
| 批量查询 | 数据库/API 用批量接口，禁止循环内单条查询（N+1 问题） |
| 数据库聚合 | 统计用 SQL 的 SUM/GROUP BY，不用代码内存遍历 |
| 缓存谨慎 | 只有测量证明瓶颈时才加缓存 |

### 11.2 安全

| 规则 | 执行指令 |
|------|----------|
| 输入校验 | 所有外部输入（用户、文件、网络）必须校验 |
| 禁止硬编码密钥 | 密钥、密码、Token 通过环境变量或密钥管理服务获取 |
| 日志脱敏 | 不打印密码、身份证号、手机号、Token |
| 最小权限 | 只申请功能必需的权限/依赖 |

## 12. 审查检查清单（Agent 交付前自检）[通用] (V2.1)

### 12.1 基础检查

- [ ] 命名是否清晰表达意图？
- [ ] 函数是否 ≤ 30 行？参数是否 ≤ 4 个？
- [ ] 类是否遵循单一职责？
- [ ] 是否有硬编码字符串/数字/配置？
- [ ] 错误处理是否完整（无空 catch）？
- [ ] 边界条件是否覆盖（null、空集合、极值）？
- [ ] 是否有遗留调试代码（console.log / print / debug 日志）？

### 12.2 架构检查

- [ ] 依赖方向是否正确（表现层 → 业务层 ← 数据层）？
- [ ] 业务层是否依赖了框架/平台代码？
- [ ] 是否有循环依赖？
- [ ] 新增的类是否注入了依赖（而非内部 new）？

### 12.3 变更影响检查

- [ ] 修改的公开 API 是否影响其他调用方？
- [ ] 新增了依赖吗？是否必要？
- [ ] 数据库 Schema 变更是否写了 Migration？
- [ ] 测试是否覆盖本次改动？

## 13. 禁止事项总表 [通用] (V3.7)

| # | 禁止 | 原因 |
|---|------|------|
| 1 | 空 catch 块 | 隐藏 Bug |
| 2 | 硬编码配置/密钥 | 环境切换出错、安全风险 |
| 3 | 巨型类（>300行）/ 巨型函数（>30行） | 难以理解和测试 |
| 4 | 复制粘贴代码 | 修改遗漏导致 Bug |
| 5 | 提交不相关改动 | 混淆历史、回退困难 |
| 6 | 浮点数存储金额 | 精度误差 |
| 7 | 循环内单条数据库查询 | N+1 性能问题 |
| 8 | 业务层依赖框架代码 | 不可移植、难以测试 |
| 9 | 生产环境 debug 日志 | 性能下降、信息泄露 |
| 10 | 无注释的 hack/workaround | 后来者不知道何时能移除 |
| 11 | 用户已提供原始素材，却自行二次加工生成衍生文件 | 偏离用户意图，原图效果无法还原（如 SplashScreen 背景图） |

## 14. 规范自我更新制度 [通用] (V3.1)

> **问题**：规范文档不是一成不变的。项目实践中会不断产生新经验，如果不主动提炼和回流，规范就会停留在"初始版本"而无法进化。本节定义**全部三份规范文档**（通用开发规范、开发阶段规划指南、项目启动指南）的自我更新机制。

### 14.1 更新触发时机

| 触发事件 | 应做的事 | 示例 |
|---------|---------|------|
| 项目问题记录中出现了**跨项目通用**的教训 | 提炼为对应规范的条目 | LazyColumn 偏移计算 → 通用开发规范附录 A 第 1 条 |
| 同类错误在不同项目中重复出现 ≥ 2 次 | 升级为规范的**禁止事项**或**强制规则** | coerceIn + 空列表反复踩坑 → 通用开发规范第 9.2 节强制规则 |
| 规范条目在实际执行中发现**不可操作**或**表述歧义** | 修改条文使其可执行，修订记录中标注"修正" | "文档必须叫 requirements.md" → 改为"需求规格内容角色" |
| 新的技术领域/平台引入了新的通用约束 | 新增章节或条目 | 引入 Room → 通用开发规范新增数据库变更规范（第 8 章） |
| 阶段规划实践中发现新的反模式或规则 | 补充到开发阶段规划指南 | 按页面拆分的后果被多次验证 → 开发阶段规划指南反模式 1 |
| 项目启动流程中发现新的遗漏模式 | 补充到项目启动指南 | 多个项目都跳过了技术栈确认 → 项目启动指南新增澄清项 |

### 14.2 沉淀标准

不是所有经验都值得进规范文档。判断依据：

| 级别 | 标准 | 去处 |
|------|------|------|
| **项目特有** | 只在当前项目/技术栈下有意义 | 留在项目的问题记录教训部分 |
| **领域通用** | 同技术栈的其他项目也可能遇到 | 进入项目编码规范的禁止事项/检查清单 |
| **跨领域通用（编码/流程类）** | 任何项目都可能遇到 | **进入通用开发规范**（附录或正文） |
| **跨领域通用（阶段规划类）** | 任何项目的阶段规划都可能遇到 | **进入开发阶段规划指南**（反模式或规则章节） |
| **跨领域通用（项目启动类）** | 任何项目的启动阶段都可能遇到 | **进入项目启动指南**（教训或规则章节） |

### 14.3 更新流程

```
1. AI 工具/人类发现问题并记录在项目的问题记录中
2. 判断经验级别（见 14.2 标准）
3. 如果是"跨领域通用" → 判断属于哪份规范文档：
   - 编码/流程/测试类 → 通用开发规范
   - 阶段规划/反模式类 → 开发阶段规划指南
   - 项目启动/需求澄清类 → 项目启动指南
4. 拟定对应规范的修改内容
5. 向人类开发者确认修改内容（规范影响面广，不可自行直接修改）
6. 确认通过 → 写入对应规范文档，更新版本号和修订记录
7. 确认不通过 → 仅保留在项目问题记录的教训部分，不提升为规范
```

**为什么需要确认**：规范文档是跨项目复用的底层文档，一条不恰当的规范比没有规范危害更大。AI 工具可以提议，但人类对业务场景的判断更可靠。

**跨文档联动**：如果一条经验同时涉及多份规范（如"测试策略未纳入阶段规划"既是阶段规划的反模式，也涉及通用规范的测试要求），应在所有相关文档中同步修改，避免一份文档改了而另一份没改。

### 14.4 作者标注规范

修订记录中的"作者"字段标注**实际使用的工具或人员名称**，不使用笼统的"AI Agent"：

| 作者值 | 含义 | 示例 |
|--------|------|------|
| 人类姓名/昵称 | 人类开发者手动修改 | `张三`、`老王` |
| AI 工具名称 | AI 工具在开发会话中修改 | `CodeBuddy`、`Trae`、`Cursor` |
| `工具名 + 人类` | AI 工具提出修改建议，人类确认后执行 | `CodeBuddy + 老王` |

目的：追溯规范的修改来源，知道是哪个工具/人在什么时候写的。不同工具的能力和风格不同，标注具体工具名比笼统的"AI Agent"更有追溯价值。

## A. 附录：经验教训速查 [通用] (V2.0)

以下是从实际项目中沉淀的通用教训，Agent 在涉及相关场景时自动查阅：

| # | 场景 | 教训 |
|---|------|------|
| 1 | 列表索引/偏移计算 | 写完必须手动验算边界值（第一项、中间项、最后项）。偏移常量定义在列表结构近处并加注释 |
| 2 | `coerceIn` + 异步数据 | 异步数据可能为空列表，`coerceIn(0, list.lastIndex)` 在空列表时抛异常。必须先 `if (list.isEmpty()) return` |
| 3 | 组件多场景复用 | 同一组件被多处调用时，参数含义差异必须在注释中显式声明（如 `todayYear=null` 代表自动生成年份范围） |
| 4 | 复制粘贴式复用 | 发现两处代码高度相似时，尽早抽取公共模块。不要等"以后再抽"，越晚成本越高 |
| 5 | 依赖包不显式声明 | 用了库中某个类的图标/工具但没声明对应依赖包，编译才暴露。引入新 API 前先确认依赖已声明 |
