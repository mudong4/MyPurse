# 项目启动指南（Agent 可执行版）

> **文档版本**：V2.1 (2026-06-17)

> **目标读者**：AI Agent（优先）与人类开发者  
> **用途**：用户给出宽泛需求时，引导澄清需求、初始化项目文档体系  
> **定位**：本指南覆盖项目生命周期的**第①阶段（需求澄清）和第②阶段（文档初始化）**。完成本指南后，进入"开发阶段规划指南"继续。  
> **版本号管理**：本文档自身的版本号管理和修订记录维护，遵循《通用开发规范》6.5 节

---

## 0. 适用范围与快速开始

### 0.1 适用范围

本指南的绝大部分内容与项目类型无关。少数带有项目特征的内容用标签标注：

| 标签 | 含义 | 说明 |
|------|------|------|
| `[通用]` | 与技术栈和项目类型无关，任何项目都应遵守 | 需求澄清流程、文档初始化顺序、AI推断标注规范 |
| `[项目经验]` | 从特定项目经验中提炼，可能带有该项目特征 | 默认技术栈推断表、具体文档模板中的技术栈字段 |

### 0.2 快速开始

当用户提出一个宽泛需求（如"做一个记账 App"、"帮我写一个工具"），按以下步骤执行：

```
1. 判断需求完整度
2. 如不完整 → 分步引导澄清（见第 1 章）
3. 如用户无法回答 → Agent 自主推断 + 标注（见第 2 章）
4. 初始化项目文档体系（见第 3 章）
5. 交给"开发阶段规划指南"继续
```

## 1. 需求澄清流程 [通用]

### 1.1 完整度判断

Agent 收到需求后，先判断是否足够支撑开发：

| 信息 | 必须程度 | 判断标准 |
|------|---------|---------|
| 核心定位 | **必须** | 给谁用？解决什么问题？一句话能说清？ |
| 平台/技术栈 | **必须** | Web/App/CLI？用什么语言/框架？ |
| 核心功能列表 | **必须** | 至少列出 3-5 个主要功能 |
| 数据存储需求 | 视项目而定 | 是否需要持久化？数据量级？ |
| 关键交互流程 | 推荐 | 核心页面的主要操作路径 |

### 1.2 分步引导策略

不要一次抛出所有问题。按优先级分步询问：

```
第一步（核心定位）：
  "这个 App 主要给谁用？解决什么核心问题？"

第二步（功能范围）：
  "核心功能有哪些？有没有明确不需要做的？"
  → 引导用户区分 P0（必须有）/ P1（应该有）/ P2（锦上添花）

第三步（技术约束）：
  "有平台偏好吗？需要离线使用吗？数据量大概多大？"
  → 如果用户没有偏好，Agent 根据功能特征推荐技术栈
```

### 1.3 常见项目类型的默认推断 [项目经验]

> 以下推断表基于当前经验覆盖的项目类型。实际项目可能不在表中，Agent 应根据项目特征自行推断，并标注 `[AI推断]`。

当用户不指定技术栈时，Agent 可根据项目类型推断：

| 项目类型 | 默认平台 | 默认技术栈 |
|---------|---------|-----------|
| 个人工具 App | Android（Kotlin + Compose） | Room + Hilt |
| 数据管理工具 | Web（React/Vue） | IndexedDB / SQLite |
| 命令行工具 | 跨平台 CLI | Python / Go |
| 微信小程序 | 微信小程序 | 云开发 |

> **注意**：以上为默认推断，应告知用户并确认。用户有权推翻。

## 2. Agent 自主补足策略 [通用]

### 2.1 何时自主补足

当用户明确表示"我不确定"或"你帮我决定"时，Agent 应主动推断。

### 2.2 补足规范

Agent 自主生成的内容必须标注来源：

| 标注 | 含义 | 示例 |
|------|------|------|
| `[AI推断]` | Agent 根据通用经验推断，未经用户确认 | `[AI推断] 使用 Material3 设计语言` |
| `[TODO: 请补充]` | Agent 无法推断，必须人工填写 | `[TODO: 请补充] 目标用户群体描述` |

### 2.3 可以自主推断的内容

- 技术选型（根据项目类型推荐）
- 文档目录结构（根据项目类型推荐）
- 通用交互模式（导航结构、表单设计等）
- 开发阶段的基础设施步骤（构建系统、依赖注入等）

### 2.4 不应自主推断的内容

- 业务规则（如"金额超过 1000 是否需要审批"）
- 用户偏好（如"喜欢暗色还是亮色主题"）
- 功能优先级（什么功能先做后做）
- 品牌/命名（项目名称、包名等）

## 3. 文档体系初始化 [通用]

### 3.1 初始化顺序

需求澄清完成后，按以下顺序创建文档：

```
1. 需求文档            → 先把需求定下来
2. 项目介绍.md          → 基于需求确定技术选型和架构
3. 开发阶段.md          → 基于需求拆分阶段
4. 开发规范.md          → 基于技术栈补充项目特定约束
5. 开发日志.md          → 空模板，记录本次初始化
6. 问题记录.md          → 空模板
```

## 4. 经验教训

> 以下教训按适用范围标注。`[通用]` 与项目类型无关，`[项目经验]` 带有特定项目特征。

### 教训 1：需求文档不能跳 [通用]

**表现**：用户说"先做起来"，跳过了需求文档，直接写代码。

**后果**：做到一半发现功能边界模糊，反复返工。后续接手的人完全不知道哪些功能是计划内的、哪些是临时加的。

**正确做法**：即使是最简版的需求文档，也必须有功能列表和优先级标注。花 10 分钟写需求文档，省 2 小时返工。

### 教训 2：文档初始化要趁早 [通用]

**表现**：项目开发了好几周才想起来补文档。

**后果**：补录开发日志和问题记录时，很多细节已经忘了。文档变成"回忆录"而非"实时记录"。

**正确做法**：需求澄清后立即初始化 6 类文档。开发日志从第一次会话就开始写。

### 教训 3：[AI推断] 标注不能省 [通用]

**表现**：Agent 推断的内容没有标注来源，用户以为是自己的决策。

**后果**：后期用户发现"我没说过要用这个技术栈"，但文档里没有区分"用户说的"和"Agent 猜的"。

**正确做法**：所有 Agent 自主生成的内容必须标注 `[AI推断]`，让用户明确知道哪些需要审核。

---

## 5. 关键规则（强制）[通用]

- **需求澄清不能跳过。** 至少要有核心定位 + 功能列表（含优先级）。
- **Agent 推断必须标注 `[AI推断]`。** 不标注视为错误。
- **文档初始化必须趁早。** 第一次会话就要建好文档骨架。
- **初始文档标注完整度。** 让后续接手的人知道哪些是已确认的、哪些是推断的。
- **自我更新遵循通用开发规范第 14 章。** 本文档的经验教训如需沉淀为规范条目，按《通用开发规范》14.2 沉淀标准判断去向，14.3 更新流程执行。
