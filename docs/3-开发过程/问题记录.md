### 问题 #1：分类管理拖拽排序性能卡顿

- **发现日期**：2026-06-15
- **严重程度**：🟡 体验问题（PC 端已可感知，移动端预期更严重）
- **所属模块**：分类管理页 (E)

**现象**：
拖拽分类条目移动时，画面有明显卡顿，不够跟手。

**根因分析**：
1. `onDrag` 回调每帧（~60fps）都会调用 `onMoveCategory`，触发 LazyColumn 重组和逐条数据库写入
2. `animateDpAsState` 为每个列表项创建动画状态对象，增加重组开销
3. 每次拖拽过程中逐条执行 `updateSortOrder`，产生大量数据库 I/O

**解决方案**：
1. `onDrag` 中仅累积 `dragOffsetY`，不做任何数据变更；实际的位置交换逻辑移至 `onDragEnd` 执行
2. 将 `animateDpAsState` 替换为简单的条件判断（`if (isDragging) 8.dp else 0.dp`）
3. 新增 `batchUpdateSortOrder` DAO 方法，使用 `@Transaction` 在单次事务中批量更新所有受影响分类的排序序号

**涉及文件**：
- `CategoryManageScreen.kt`：拖拽手势逻辑重构
- `CategoryManageViewModel.kt`：`onMoveCategory` 改为调用批量更新
- `CategoryDefDao.kt`：新增 `batchUpdateSortOrder`
- `CategoryRepository.kt` / `CategoryRepositoryImpl.kt`：新增批量更新接口及实现

### 问题 #2：拖拽排序后点击位置错乱

- **发现日期**：2026-06-15
- **严重程度**：🟠 功能缺陷
- **所属模块**：分类管理页 (E)

**现象**：
拖拽"餐饮"到新位置后，再次点击"餐饮"，响应的是原来位置（拖拽前的位置）的分类。

**根因分析**：
1. **闭包捕获了旧值**：`LazyColumn` 中 `itemsIndexed` 的 `pointerInput(category.id)` 的 key 未因重排序而改变，导致手势检测回调中捕获的 `index` 和 `categories` 仍是拖拽前的旧值
2. **UI 状态被覆盖**：`loadCategories()` 中的 `collect` 使用了 `_uiState.value.copy(...)` 而非 `current.copy(...)`，导致 `expandedParentId` 等 UI 状态字段在数据刷新时被重置为默认值

**解决方案**：
1. 在 `itemsIndexed` 的 lambda 中使用 `rememberUpdatedState` 包装 `index` 和 `categories`，确保长生命周期的 gesture 回调始终读取最新值
2. 将 `_uiState.value.copy(...)` 改为 `current.copy(...)`，保留所有 UI 状态字段

**涉及文件**：
- `CategoryManageScreen.kt`：添加 `rememberUpdatedState`，修复手势回调中的闭包捕获
- `CategoryManageViewModel.kt`：修复 `loadCategories()` 中的状态合并方式

**教训**：
- Compose 中 `pointerInput` 的 key 不变时，内部 lambda 捕获的变量不会自动更新，必须用 `rememberUpdatedState` 桥接
- `StateFlow.collect` 中使用 `value.copy()` 会丢弃 collect 之前的并发更新，应使用 `current.copy()`

### 问题 #3："恢复默认"按钮无响应

- **发现日期**：2026-06-15
- **严重程度**：🟠 功能缺陷
- **所属模块**：分类管理页 (E)

**现象**：
1. 点击"恢复默认"按钮后，分类顺序未恢复，UI 无变化
2. 按钮仅恢复支出侧还是同时恢复收入侧不明确

**根因分析**：
1. `restoreDefaultCategories` 方法只处理了"添加缺失的一级分类"，完全忽略了：
   - 已存在分类的 `sortOrder` 恢复为默认值
   - 子分类的恢复
2. 子分类数据（`CategoryDefaults.subCategories`）从未被使用

**解决方案**：
重写 `restoreDefaultCategories` 方法，执行以下三步：
1. 将已存在的一级分类的 `sortOrder` 重置为默认值
2. 添加缺失的一级分类（含正确的 `sortOrder`）
3. 遍历 `CategoryDefaults.subCategories`，添加缺失的子分类

**涉及文件**：
- `CategoryRepositoryImpl.kt`：重写 `restoreDefaultCategories`

**待确认**：
- 用户提问"恢复默认是把收入和支出都恢复吗"——当前实现仅恢复当前选中的那一侧（支出或收入），通过顶部 Tab 切换后需分别点击恢复。后续可考虑增加"恢复全部"选项。

### 问题 #4：分类模式下 LazyColumn 重复 Key 导致闪退

- **发现日期**：2026-06-16
- **严重程度**：🔴 阻塞
- **所属模块**：流水列表页 (D)

**现象**：
在流水列表底部工具栏，将分组方式从"时间"切换到"一级分类"或"二级分类"后，App 立即闪退。

**根因分析**：
`GroupedTransactionList` 中，LazyColumn 的 item key 始终使用 `group.rangeStart.toString()`。在分类模式下，所有 `TimeGroup` 的 `rangeStart` 都是同一个时间范围起始时间戳（如 `1718409600000`），导致多个 group header 产生相同的 key `"header_1718409600000"`，LazyColumn 检测到重复 key 后抛出异常崩溃。

**解决方案**：
根据分组模式区分 key 生成策略：
- 时间模式（`GroupMode.TIME`）：继续使用 `rangeStart.toString()`，因为每个时间段的起始时间天然唯一
- 分类模式（`CATEGORY_L1` / `CATEGORY_L2`）：改用 `group.label`（如"餐饮"、"交通"），因为分类名天然唯一

修改位置：`TransactionListScreen.kt` 的 `GroupedTransactionList` 函数，将 key 生成逻辑改为：
```kotlin
val groupKey = if (groupMode != GroupMode.TIME) group.label else group.rangeStart.toString()
```

**涉及文件**：
- `TransactionListScreen.kt`：修复 `GroupedTransactionList` 中的 `groupKey` 生成逻辑

**教训**：
- 当分组依据不是时间时，不能假设 `rangeStart` 唯一
- LazyColumn 的 `item(key = ...)` 的 key 必须在整个列表范围内唯一，重复 key 会导致崩溃而非警告

### 问题 #5："StandaloneCoroutine was cancelled" 系统 Toast

- **发现日期**：2026-06-16
- **严重程度**：🟡 体验问题（仅 debug 构建出现，release 不受影响）
- **所属模块**：全局 / 流水列表页 (D)

**现象**：
操作过程中偶尔弹出 Toast 提示 "StandaloneCoroutine was cancelled"，不影响功能但观感不佳。

**根因分析**：
Android 14+ 系统在 debug 构建中会检测到 `CancellationException` 并通过 Toast 展示。本项目 `TransactionListViewModel` 中 `loadData()` 和 `searchData()` 使用 `loadJob?.cancel()` 取消上一个加载协程，当用户快速切换筛选条件时触发 cancel，系统将其作为"未捕获异常"弹出 Toast。

**解决方案**：
1. `TransactionListViewModel` 中：`cancel()` 前先注册 `invokeOnCompletion` 空回调吸收异常；协程内部显式 catch `CancellationException` 并重新抛出（遵循协程规范），与普通异常分开处理
2. `MyPurseApplication` 中：设置全局 `UncaughtExceptionHandler`，对 `CancellationException` 静默处理（仅打 log），其他异常仍走原有崩溃流程

**涉及文件**：
- `TransactionListViewModel.kt`：`loadData()`、`searchData()` 的 cancel 和异常处理逻辑
- `MyPurseApplication.kt`：新增全局未捕获异常处理器

**补充说明**：
正常的 release 应用不会出现此提示，因为该检测是 Android 系统仅在 debuggable=true 的构建中启用的调试行为。但即便如此，处理方案本身（全局异常处理器 + 区分 CancellationException）也是 Android 开发的最佳实践。

### 问题 #17：统计页 WheelPickerForStat 是公共 WheelPicker 的复制品

- **发现日期**：2026-06-17
- **严重程度**：🟡 体验问题（功能正常但存在维护风险：公共 WheelPicker 修 bug 时统计页副本不会同步）
- **所属模块**：统计页 (C) / 公共组件

**现象**：
统计页 `YearMonthSheet` 中使用的 `WheelPickerForStat` 与 `ChineseDatePickerDialog.kt` 中的 `WheelPicker` 功能完全一致，但实现有差异：
- 公共版使用 `derivedStateOf`（同步），副本使用 `snapshotFlow`（有帧延迟）
- 公共版有 `spacerCount` 偏移量计算和空列表防护，副本没有
- 公共版修了多次 bug 后，副本仍停留在初始实现状态

**根因分析**：
统计页的时间选择器是独立开发的一套，没有复用 `ChineseDatePickerDialog` 中已有的滚轮组件。这是"同功能同组件"原则缺失的典型表现。

**解决方案**：
1. 公共 `WheelPicker` 从 `private` 改为 `internal`，新增 4 个颜色参数（`highlightColor`、`selectedTextColor`、`unselectedTextColor`、`backgroundColor`）
2. 删除统计页 `WheelPickerForStat`（~100 行），改用公共 `WheelPicker`，传入统计页配色参数
3. 统计页滚轮同时获得了公共版的改进：无帧延迟、正确偏移量、空列表防护

**涉及文件**：
- `ui/components/ChineseDatePickerDialog.kt`：`WheelPicker` 可见性提升 + 参数化颜色
- `ui/statistics/StatisticsScreen.kt`：删除 `WheelPickerForStat`，改用公共 `WheelPicker`

**教训**：
- 组件复用不通过参数控制差异，就会导致"复制一份改改"——这是技术债务
- 扫描确认：项目中还有年份选择器重复（统计页 3 处 + 流水列表 1 处各自实现），待后续处理

### 问题 #6：Room 版本升级导致已有数据丢失

- **发现日期**：2026-06-16
- **严重程度**：🔴 阻塞
- **所属模块**：数据层（AppDatabase）

**现象**：
添加预算功能后（数据库 version 从 2 升到 3），重新运行 App，之前添加的所有记录和分类自定义全部丢失。

**根因分析**：
`AppDatabase` 的 `fallbackToDestructiveMigration()` 策略在检测到版本号变化且无 Migration 时，会**直接删除旧数据库并重建**。数据库版本从 2→3 未提供对应的 Migration，触发了破坏性迁移。

**解决方案**：
1. 短期：为版本 2→3 添加正确的 Migration（`addMigrations(MIGRATION_2_3)`）
2. 长期原则：每次数据库 schema 变更都必须提供对应 Migration；`fallbackToDestructiveMigration()` 仅作为最后兜底

**涉及文件**：
- `AppDatabase.kt`：需补充 MIGRATION_2_3 和 MIGRATION_1_2

**教训**：
- `fallbackToDestructiveMigration()` 在开发期看似方便，但一旦有真实数据就会造成灾难性后果
- 每次加表/改字段都应写 Migration，不能依赖 destructive fallback
- 开发阶段最好也接入导出备份，减少数据丢失风险

### 问题 #9：月粒度趋势图导航到历史时间时显示错误数据

- **发现日期**：2026-06-16
- **严重程度**：🟠 功能缺陷
- **所属模块**：统计页趋势图 (C)

**现象**：
在趋势图中将粒度切换为"月"，然后导航到 2022 年（或其他历史年份），图表没有显示"无数据"，而是错误地显示了 2025 年（数据库中最早有数据的年份）的数据。

**根因分析**：
月粒度趋势查询 `getMonthlyExpenseTrendOnce` 的 SQL 只有下限 `date >= :sinceTimestamp`，缺少上限 `date <= rangeEnd`。其他粒度（日/周/季度）都正确使用了 `BETWEEN rangeStart AND rangeEnd`，唯独月粒度遗漏了上限条件。

当用户导航到 2022 年 3 月时，`rangeStart = 2021-04-01`，SQL 查出来的是 `>= 2021-04-01` 的所有数据。因为数据库中最早的数据是 2025 年的，图表就显示了 2025 年的数据，而不是期望的空结果。

**解决方案**：
1. DAO 层：`getMonthlyExpenseTrendOnce(sinceTimestamp)` 改为 `getMonthlyExpenseTrendOnce(rangeStart, rangeEnd)`，SQL 从 `WHERE date >= :sinceTimestamp` 改为 `WHERE date BETWEEN :rangeStart AND :rangeEnd`
2. Repository 层：调用时传入 `rangeEnd` 参数

**涉及文件**：
- `TransactionDao.kt`：DAO 方法签名和 SQL
- `TransactionRepositoryImpl.kt`：调用处补传 rangeEnd

**教训**：
- 时间范围查询必须同时设置上下限，单边条件在大时间跨度下会产生静默错误（不崩溃但数据错误，更难发现）
- 同类粒度的查询应保持一致的 SQL 模板，不一致往往是 bug 的信号

### 问题 #10：BudgetEntity 架构回退（按月存储 → 单条记录）

- **发现日期**：2026-06-16
- **严重程度**：🟠 功能缺陷
- **所属模块**：预算功能 (F)

**现象**：
最初预算设计使用 `year + month` 复合主键（每个月的预算独立存储）。实际使用中发现该方案存在严重问题：首页展示时无法确定取哪个预算值、用户需每月重新设置预算、跨月查询逻辑复杂。

**涉及文件**：
- `BudgetEntity.kt`：字段从 year+month+amount 简化为 id+amount
- `BudgetDao.kt`：查询改为 `WHERE id = 1`，写入用 `OnConflictStrategy.REPLACE`

**教训**：
- 个人记账场景下，按月预算的复杂度收益不成比例
- 过度设计（预先假设需要历史预算追踪）增加了不必要的复杂度

### 问题 #11：周日周范围计算边界错误

- **发现日期**：2026-06-16
- **严重程度**：🟡 体验问题
- **所属模块**：流水列表页 (D) / 首页 (A)

**现象**：
周日时，本周范围（周一至周日）计算错误。`Calendar.SUNDAY = 1`，`Calendar.MONDAY = 2`，直接用 `dayOfWeek - Calendar.MONDAY` 计算时周日会得到 `-1`，导致本周起始日期计算到错误的日期。

**解决方案**：
对周日做特殊处理：`val daysFromMonday = if (dayOfWeek == Calendar.SUNDAY) 6 else dayOfWeek - Calendar.MONDAY`。周日时向前推 6 天到周一。

**涉及文件**：
- `TransactionRepositoryImpl.kt`：`getWeeklySummary` 中的周范围计算

**教训**：
- `Calendar` 的周日=1、周一=2 的常量定义容易造成直觉偏差
- 边界条件（一周的起始和结束日）需要显式测试

### 问题 #12：收入聚合用 `!= '支出'` 而非 `= '收入'`

- **发现日期**：2026-06-16
- **严重程度**：🟡 体验问题
- **所属模块**：数据层 (TransactionDao)

**现象**：
余额查询和汇总查询中，收入聚合使用 `WHERE flow_type != '支出'` 而非 `WHERE flow_type = '收入'`。正常设计应直接用等值匹配。

**根因分析**：
系统中可能存在或未来可能引入非标准的正向流水类型（如"退款"、"转账收入"等）。使用 `= '收入'` 会遗漏这些类型的金额，导致余额和汇总数值偏小。`!= '支出'` 是兜底策略——所有非支出都视为收入。

**当前状态**：
该写法是有意为之的防御性设计，当前功能正常。但语义上不够明确，如果将来引入第三种类型（如"内部转账"既不增也不减），当前写法会错误地将其计入收入。

**涉及文件**：
- `TransactionDao.kt`：`getBalance`、`getDaySummary`、`getRangeSummary` 等查询

### 问题 #14：WheelPicker 滚轮选择器高亮条与选中文字错位

- **发现日期**：2026-06-17
- **严重程度**：🟡 体验问题
- **所属模块**：日期选择器组件（ChineseDatePickerDialog）

**现象**：
1. 年月滚轮选择器（`YearMonthWheelSheet`）中，灰色高亮条与蓝色选中文字错位一行——高亮条在某项上，蓝色文字在相邻的另一项上
2. 流水列表页的日期选择器在打开时闪退

**根因分析**：
1. **`spacerCount` 值错误**：LazyColumn 结构为 `[顶部spacer(80dp)] + [items列表] + [底部spacer(80dp)]`，只有一个顶部 spacer item，但 `spacerCount` 被设为 `2`，导致所有 LazyColumn 全局索引 → items 数组索引的偏移计算差了一行
2. **空列表 `coerceIn` 崩溃**：流水列表的 `availableYears` 是异步加载的，初始状态 `yearList` 为空列表 `[]`（非 null）。`items.size - 1` 计算得 `-1`，`coerceIn(0, -1)` 抛出 `IllegalArgumentException`
3. **`snapshotFlow` 延迟**：基于 `snapshotFlow` 的居中项检测在快速滚动时有帧延迟，改用 `derivedStateOf` 实现实时计算

**解决方案**：
1. 将 `spacerCount` 从 `2` 修正为 `1`，并调整所有索引计算公式：
   - `initialFirstVisibleIndex = spacerCount + selectedIndex - visibleItems/2`
   - `animateScrollToItem((index + spacerCount - visibleItems/2).coerceAtLeast(0))`
   - 居中项计算：`itemsIdx = nearestIdx - spacerCount`
2. 将 `snapshotFlow` + `collect` 替换为 `derivedStateOf`，在组合阶段同步计算居中项
3. 为 `WheelPicker` 添加 `if (items.isEmpty()) return` 空列表防护
4. `resolvedYearList` 中 `yearList?.sortedDescending()` 改为 `yearList?.takeIf { it.isNotEmpty() }?.sortedDescending()`，过滤空列表

**涉及文件**：
- `ChineseDatePickerDialog.kt`：修复 `spacerCount`、`derivedStateOf` 重构、空列表防护、`resolvedYearList` 过滤

**教训**：
- 涉及 LazyColumn 索引映射时，必须精确数清楚前面有几个 spacer/header item，写成常量并加注释说明
- 任何涉及"偏移量"的计算（列表索引、像素位置、数组下标），写完一定要手动验算几个边界值（第一项、中间项、最后项）
- `snapshotFlow` 有延迟，需要帧级同步的场景应使用 `derivedStateOf`
- 异步加载的列表数据在初始状态可能为空（非 null），`coerceIn(0, items.lastIndex)` 在 `items` 为空时 `lastIndex` 为 `-1`，会导致崩溃。应始终先判空

**举一反三**：
- 统计页的 `YearMonthSheet` 中有 `WheelPickerForStat`（`WheelPicker` 的副本），如果存在类似的 `spacerCount` 错误，也应一并修复
- 记一笔（`AddTransactionScreen`）和流水列表（`TransactionListScreen`）共用同一个 `ChineseDatePickerDialog`，但传入的 `yearList` 参数不同（前者传 null 自动生成 ±30 年，后者从数据库异步统计有数据的年份），修改时需保证两种场景都正常

### 问题 #18：Git 回退操作导致 UI 代码大量丢失

- **发现日期**：2026-06-17
- **严重程度**：🔴 阻塞（代码层面：设置页回到占位版、路由被还原、已删除文件被恢复）
- **所属模块**：全局（Git 操作失误 + 缺乏防护机制）

**现象**：
1. 会话 #14 执行 `git checkout` 回退操作后，多个 UI 文件被还原到旧版本
2. `SettingsScreen.kt` 从完整的 4 分组页面退回到占位版（只有"设置"标题）
3. `Route.kt` 被还原到 HEAD，丢失了 `AddTransaction` 的 `transactionId` 参数
4. 已删除的 `TransactionDetailScreen.kt` 等 3 个文件被恢复
5. 颜色体系在回退过程中出现偏差——部分文件是会话 #14 的新配色，部分是被还原的旧配色，导致视觉不一致

**根因分析**：
1. **直接原因**：`git checkout` 操作覆盖了工作区中未提交的改动。项目自 `8dddcf0` 提交后，阶段 7/8/9 的 25 个修改文件 + 6 个删除文件 + 14 个新增文件从未提交过 git，`checkout` 命令将这些未追踪的改动全部丢弃
2. **深层原因 — 缺乏提交节奏**：多个阶段的大量改动集中在一个工作区中，中间没有提交 checkpoint。一旦出错，丢失的是数天的工作成果
3. **深层原因 — 缺乏回退前的安全检查**：执行 `git checkout` 前没有确认 `git status` 中哪些文件会被影响，也没有做备份（`git stash` 或手动复制）
4. **深层原因 — 文档状态与代码状态不同步**：开发日志记录了会话 #7~#9 的所有改动，但 `git checkout` 回退后，代码状态与会话记录产生了偏差。后续会话只能根据文档记录手动重建，但重建过程中引入了新的不一致（如通配符导入、颜色值微调等）

**修复过程**：
会话 #16 耗时整个会话手动修正回退偏差：
1. `SettingsScreen.kt`：按设计文档重建（4 分组、数据管理含清除、显示设置灰显、关于含反馈与建议）
2. `Route.kt`：补回 `AddTransaction` 的 `transactionId` 参数，删除 `TransactionDetail` 路由
3. `MyPurseNavHost.kt`：移除 RecurringTemplate/TransactionDetail 的 composable 注册，修正编辑跳转

**但修复后仍有残留问题**：
- `SettingsScreen.kt` 通配符导入 `import icons.filled.*` 导致 IDE 警告（后续修复为显式导入）
- 颜色风格与会话 #14 之前有微妙差异（`surfaceVariant`、`outlineVariant`、`windowBackground` 等被显式赋值）
- 部分回退偏差可能尚未被发现

**涉及文件**：
- `ui/settings/SettingsScreen.kt`：完全重建
- `ui/navigation/Route.kt`：补回丢失的改动
- `ui/navigation/MyPurseNavHost.kt`：移除过期路由注册
- `ui/theme/Color.kt`、`ui/theme/Theme.kt`、`ui/theme/AppColors.kt`：颜色体系存在版本不一致

**教训**：
- **Git 是唯一的防线**：在 AI Agent 阶段式开发模式下，多个独立会话的改动共享同一个工作区。如果没有定期提交，一次 git 误操作就会丢失跨会话的累积成果。这是比单人开发更脆弱的状态
- **提交粒度应与会话边界对齐**：每个阶段/会话完成后应立即 commit，形成 checkpoint。不攒多个阶段再一起提交
- **`git checkout -- <file>` 的风险被严重低估**：当前规范 5.3 节标注为"低"风险，但这是错误的——在工作区有大量未提交改动时，`checkout --` 同样不可逆。规范应修正
- **回退操作前必须做快照**：执行任何可能丢失改动的 git 操作前，应先用 `git stash` 或 `git diff > backup.patch` 保存当前状态
- **文档不能替代 git**：开发日志虽然记录了改动内容，但无法替代代码本身。手动从文档重建代码必然引入偏差（如通配符导入、颜色值差异等）
- **颜色/主题改动应在独立 commit 中完成**：颜色体系从 1 层扩展到 3 层（`Color.kt` → `AppColors.kt` → 页面级色值），涉及多个文件联动。如果在回退中这些文件的版本不一致，就会出现"颜色看起来不对"的问题

**举一反三**：
- 项目中所有未提交的改动（25 个修改文件 + 6 个删除 + 14 个新增）都处于风险中，应尽快提交
- 规范 5.3 节的 `git checkout --` 风险等级应从"低"改为"中高"，并增加前置检查步骤
- 应在开发流程中增加"阶段完成 → 立即 commit"的强制要求

### 问题 #16：编辑流水记录页日历未中文化 + 代码重复

- **发现日期**：2026-06-17
- **严重程度**：🟠 功能缺陷（日历未中文化）+ 🟡 架构问题（代码重复）
- **所属模块**：流水列表页 (D) / 编辑详情页

**现象**：
1. 流水列表中点击单条记录进入编辑页，日期选择器显示的是 Material3 原生英文日历（Jan/Feb/Sun/Mon），而非中文日历
2. `TransactionDetailScreen` 独立实现了一套编辑 UI（流水类型选择、金额输入、备注、日期选择），与 `AddTransactionScreen` 功能高度重叠，且功能有缺失（不能修改分类）

**根因分析**：
- `TransactionDetailScreen` 第 153 行使用了 `DatePickerDialog`（Material3 原生），而记一笔和流水列表用的是 `ChineseDatePickerDialog`
- 编辑页被设计为独立页面而非复用记一笔，导致分类选择器、金额键盘等全部重新实现了一遍

**解决方案**：
1. 删除 `TransactionDetailScreen`、`TransactionDetailViewModel`、`TransactionDetailUiState` 三个文件
2. `AddTransactionScreen` 增加 `transactionId` 参数，支持编辑模式
3. 编辑模式下：标题改为"编辑记录"、隐藏"保存并继续"按钮、显示模板解除关联警告
4. 编辑导航从 `Route.TransactionDetail(id)` 改为 `Route.AddTransaction(transactionId = id)`
5. 日期选择器自然继承 `ChineseDatePickerDialog`，日历中文化问题自动修复

**涉及文件**：
- 删除：`TransactionDetailScreen.kt`、`TransactionDetailViewModel.kt`、`TransactionDetailUiState.kt`
- 修改：`AddTransactionScreen.kt`、`AddTransactionViewModel.kt`、`AddTransactionUiState.kt`、`Route.kt`、`MyPurseNavHost.kt`

**教训**：
- 遇到"编辑/新建"这类功能时，优先考虑复用而非新建。编辑本质上是"预填数据的新建"，应在同一页面通过模式切换实现
- 同一个功能（日期选择）出现在多处时，必须确保使用同一组件，避免每处独立实现导致行为不一致
- 举一反三：`RecurringTemplateEditScreen` 也可能存在类似问题（编辑模板 vs 新建模板），应检查是否需要合并

**架构层面教训**（已沉淀为开发规范 2.3）：
- **组件归属错误**：公共组件放在页面目录下，被其他页面开发者误判为"私有组件"。凡被 ≥2 个页面使用的组件必须放在 `ui/components/`
- **缺少组件索引**：项目没有"已有公共组件清单"，新页面开发者不知道已有什么轮子。应在文档中维护清单，并在开发前强制搜索
- **阶段式开发的天然盲区**：每个阶段独立会话缺乏信息传递，需要通过文档机制弥补。详见 `开发规范.md` 2.3 节
