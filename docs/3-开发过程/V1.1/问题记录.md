### V1.1-#1：统计图配色功能实际无可见效果 → 移除

- **发现日期**：2026-06-21
- **严重程度**：🟢 优化建议
- **所属模块**：设置页 / 统计页
- **修复**：✅ 已修复（2026-06-21，会话 #45，移除功能 + 旧自定义分类颜色回填）

**现象**：
V1.1 新增的「统计图配色」自定义功能在设置页有入口，用户可编辑 8 色调色板，但实际图表中几乎看不到效果。原因：每个分类在 V1.0.1（11d）已有了自己的 `color` 字段，`categoryColor()` 优先返回分类色，`chartPalette` 只在分类 color=0 时才作为 fallback 生效。内置分类全有色，新自定义分类自动配色，chartPalette 几乎没有被使用。

**解决方案**：
1. 移除统计图配色入口（设置页）、编辑器 BottomSheet、相关 DataStore 键和 JSON 序列化函数（10 个文件）
2. 新增 `backfillCustomCategoryColors()` 给旧自定义分类（color=0）一次性补色，确保所有分类都有稳定颜色

**涉及文件**：
- 移除：`SettingsScreen.kt`、`SettingsUiState.kt`、`SettingsViewModel.kt`、`MainActivity.kt`、`Theme.kt`、`StatisticsScreen.kt`、`HomeScreen.kt`、`AppColors.kt`、`UserPreferencesRepository.kt`
- 删除：`ChartPaletteEditorSheet.kt`
- 新增逻辑：`DatabaseInitializer.kt` → `backfillCustomCategoryColors()`

**教训**：
- 功能开发前应先评估"对已有功能的增量价值"，而非"独立功能本身是否完成"。统计图配色在 11d 分类颜色体系建立后价值几乎为零
- 需求文档中功能与已有功能的交互（如优先级链）应在设计阶段就显式分析

### V1.1-#2：固定收支模板收入分类选择显示为支出分类

- **发现日期**：2026-06-21
- **严重程度**：🟠 功能缺陷
- **所属模块**：固定收支模板 (G)
- **修复**：✅ 已修复（2026-06-21，会话 #45，三处 sign 修正）

**现象**：
在固定收支模板中创建或编辑收入模板时，分类选择器显示的是支出类别而非收入类别。

**根因分析**：
`TemplateViewModel.kt` 三处问题：
1. `loadForCreate()`：`getTopLevelCategoriesBySignOnce(-1)` 硬编码支出 sign
2. `loadForEdit()`：同上，编辑收入模板时同样加载支出分类
3. `onFlowTypeChanged()`：切换收支类型时只更新 `flowType` 字段，不重新加载分类列表

**解决方案**：
1. `FlowTypeOption` 新增 `sign` 属性（EXPENSE=-1, INCOME=1）
2. `loadForCreate()` → 使用 `FlowTypeOption.EXPENSE.sign`（语义清晰）
3. `loadForEdit()` → 使用模板实际 `flowType.sign` 加载对应分类
4. `onFlowTypeChanged()` → 重新加载对应 sign 的分类列表，清空已选分类

**涉及文件**：
- `ui/template/TemplateUiState.kt`：`FlowTypeOption` 新增 `sign` 参数
- `ui/template/TemplateViewModel.kt`：三处 `sign` 修正

**教训**：
- `-1` 这样的魔术数字不要硬编码在多处，应使用枚举/常量的属性
- 下拉/切换控件改变数据时，不仅要更新 UI 状态字段，还要考虑是否有**关联数据需要重新加载**（如切换流水类型 → 分类列表变化）

### V1.1-#3：环形图点击检测不准 → 简化为仅图例行点击

- **发现日期**：2026-06-21
- **严重程度**：🟡 体验问题
- **所属模块**：统计页环形图 (C)
- **修复**：✅ 已修复（2026-06-21，会话 #45，移除 Canvas pointerInput 点击检测，保留图例行 clickable）

**现象**：
环形图同时支持两种选中交互：点击环形色块 + 点击下方图例行。饼图角度检测不准（`atan2` 角度计算 + 环形命中区域判断有边缘误差），且两个入口做同一件事是冗余设计。

**解决方案**：
移除 Canvas 的 `pointerInput` + `detectTapGestures` 角度计算块（约 45 行），只保留图例行的 `.clickable` 作为唯一的选中交互入口。Canvas 恢复为简单的绘制逻辑（inline sweep 计算）。

**涉及文件**：
- `ui/statistics/StatisticsScreen.kt`：DonutChart 组件简化

**教训**：
- 命中检测（hit testing）在自定义 Canvas 中实现容易有边界误差，优先用 Compose 原生 `.clickable` 等修饰符
- 同一个操作提供多个入口时，如果其中一个不准，应果断砍掉而非修补——冗余入口增加维护成本

### V1.1-#4：`colorToHex` 导入路径错误导致编译失败

- **发现日期**：2026-06-21
- **严重程度**：🔴 阻塞（编译不通过）
- **所属模块**：V1.1 主题切换 — 11c 颜色选择器
- **修复**：✅ 已修复（2026-06-21，会话 #44，删除错误的跨包导入行）

**现象**：
`ColorPickerSheet.kt` 编译报错 `Unresolved reference 'colorToHex'`。

**根因分析**：
`colorToHex()` 定义在 `ColorWheel.kt`（包 `com.wyd.mypurse.ui.components`），与 `ColorPickerSheet.kt` 同包。但 Agent 在创建 `ColorPickerSheet.kt` 时错误地写了 `import com.wyd.mypurse.ui.theme.colorToHex`，导入路径指向不存在的包。

**解决方案**：
删除错误的 import 行。同包下的顶层函数无需显式导入，Kotlin 编译器自动可见。

**涉及文件**：
- `ui/components/ColorPickerSheet.kt`：删除 `import com.wyd.mypurse.ui.theme.colorToHex`

**教训**：
- 同包下新建文件引用已有顶层函数时，应先确认目标函数的所在包，而非凭记忆写入 import
- Kotlin 同包可见性规则：同包内的顶层函数、变量可直接引用，无需 import

### V1.1-#6：RecurringScheduler 执行无 UI 反馈，首页数字闪变

- **发现日期**：2026-06-22
- **严重程度**：🟡 体验问题
- **所属模块**：自动记账引擎 (RecurringScheduler)
- **修复**：✅ 已修复（2026-06-22，会话 #48，splash 等 execute() 完成后再隐藏，保证至少 800ms）

**现象**：
用户打开 App 后 splash 消失、首页出现，但首页数字（本月已花、余额等）会闪烁变化——因为自动记账引擎（RecurringScheduler.execute()）在后台逐天补记固定收支流水，首页渲染时补记可能才执行到一半。月初 1 号（需补记的模板条目多）时尤其明显：¥0 → ¥25 → ¥3025 → ¥3100，数字在 1-2 秒内跳数次。

**根因分析**：
`MainActivity.kt` 中 splash 隐藏逻辑与 `scheduler.execute()` 完全独立：

```kotlin
LaunchedEffect(Unit) {
    scheduler.execute()  // fire-and-forget
}
// splash 隐藏只依赖时间条件，不等 execute 完成
```

首页渲染时 automatic 记账还在后台执行，Room 响应式更新导致 UI 数字逐次刷新。

**解决方案**（建议）：
方案 A（简单）：让 splash 等 `execute()` 返回后再隐藏（可能多卡 1-2 秒）
方案 B（细腻）：splash 按现有速度消失，但在首页加轻量"正在同步固定收支…"提示条

**涉及文件**：
- `MainActivity.kt`：`AppEntry()` 中的 LaunchedEffect
- `RecurringScheduler.kt`：无任何 StateFlow 暴露执行状态

**教训**：
- 后台数据写入若影响首屏展示的数据，必须有 loading 状态反馈，否则用户看到的是"跳动的不准确数据"
- splash 的生命周期应与数据就绪状态绑定，而非仅与时间挂钩

### V1.1-#7：RecurringScheduler 逐条写入无事务包裹，中断时状态不一致

- **发现日期**：2026-06-22
- **严重程度**：🟡 体验问题（有防重兜底，实际未触发出错）
- **所属模块**：自动记账引擎 (RecurringScheduler)
- **修复**：✅ 已修复（2026-06-22，会话 #48，先收集所有待插入记录 → 批量 insertTransactions（单事务）→ 再更新模板状态）

**现象**：
`execute()` 对每个模板逐天循环，每发现一个需补记的日期就调用 `transactionRepository.insertTransaction()` 逐条写入。若写入中途 App 被杀死（如用户划掉、系统回收），已写入的流水已落库，但模板的 `lastExecutedDate` 未更新——下次启动需重新遍历已写入的日期，依赖防重检查（`countByTemplateAndDate()`）跳过已有记录。

**根因分析**：
```kotlin
// RecurringScheduler.kt 第 100-110 行
while (calendar.timeInMillis <= today) {
    if (matchesTemplate(...)) {
        // 逐条 insertTransaction，无事务包裹
        transactionRepository.insertTransaction(...)
    }
    calendar.add(Calendar.DAY_OF_MONTH, 1)
}
// lastExecutedDate 在所有日期遍历完后才更新
```

没有 `@Transaction` 批量 DAO 方法，没有用 `withTransaction {}` 包裹。最坏情况：5 个模板 × 90 天 × 2 次 DB 操作（写 + 防重查）= 900 次独立 DB 操作。

**解决方案**（建议）：
DAO 层新增 `@Transaction insertTransactions(List<TransactionEntity>)` 批量插入方法，在一次事务中完成所有写入 + `lastExecutedDate` 更新。

**涉及文件**：
- `data/local/RecurringScheduler.kt`：execute() 逻辑
- `data/local/dao/TransactionDao.kt`：新增批量插入方法

**教训**：
- 循环内的逐条 DB 写入应有事务包裹，保证原子性
- 目前靠防重机制兜底未出错，但不应长期依赖单点防御

### V1.1-#8：固定收支模板月/年周期 day 输入超界无提示

- **发现日期**：2026-06-22
- **严重程度**：🟡 体验问题
- **所属模块**：固定收支模板 (G)
- **修复**：✅ 已修复（2026-06-22，会话 #48，MONTHLY 和 YEARLY day 输入框下方加灰色提示文字）

**现象**：
创建/编辑月度模板时，用户可输入"每月 31 号"。2/4/6/9/11 月没有 31 号，系统静默退到 30 号（或 28 号）。年度模板同理——用户可输入"2 月 30 日"，系统退到 2 月 28 日。用户不知情，可能困惑"为什么我的 31 号记录从来没出现过"。

**根因分析**：
调度层 `matchesMonthlyDay()` 已用 `coerceAtMost(lastDay)` 兜底，不会崩溃也不会丢数据。但模板编辑页面（`TemplateEditScreen.kt`）的输入框只有 `coerceIn(1, 31)` 范围限制，没有任何提示告知用户月末截断行为。

**解决方案**（建议）：
在月度 day 输入框下方加灰色提示文字："若当月无此日期，自动退到月末最后一天"。年度同理。

**涉及文件**：
- `ui/template/TemplateEditScreen.kt`：MONTHLY 和 YEARLY 的 day 输入框

**教训**：
- 输入校验的边界行为（如截断、回退）如果对用户不可见，就是隐藏的设计规则。应通过提示文字暴露
- 调度层的容错逻辑（`coerceAtMost`）不应成为唯一的正确性保证——UI 层也应引导用户理解边界

### V1.1-#9：首页月支出收入显示为零（getMonthlySummary 日期范围错误）

- **发现日期**：2026-06-23
- **严重程度**：🔴 阻塞
- **所属模块**：首页概览 (A)
- **修复**：✅ 已修复（2026-06-23，会话 #46，getMonthlySummary() 中 Calendar 重置 DAY_OF_MONTH 为 1）

**现象**：
首页概览图片和"本月支出/收入"卡片显示为零，但实际本月有支出记录。记一笔新账后，只显示新记的金额，旧数据仍然不被计入。

**根因分析**：
`TransactionRepositoryImpl.getMonthlySummary()` 中，`Calendar.getInstance()` 保留了当天的 `DAY_OF_MONTH`。代码只设置了 `YEAR` 和 `MONTH`，没有把 `DAY_OF_MONTH` 重置为 1。导致 `monthStart` 计算为"当月今天 00:00:00"而非"当月 1 号 00:00:00"。
例如今天是 6 月 23 日，SQL 查询范围变成 6月23日~6月30日，1-22 日的旧数据全部被排除。

**对比**：`getYearlySummary()` 没有此问题，因为它在 `getDayStart()` 前显式执行了 `set(Calendar.DAY_OF_MONTH, 1)`。

**解决方案**：
在 `getMonthlySummary()` 的 `Calendar` 初始化中，`getDayStart(cal)` 调用之前加一行 `set(Calendar.DAY_OF_MONTH, 1)`。

**涉及文件**：
- `data/repository/TransactionRepositoryImpl.kt`：`getMonthlySummary()` 方法，新增 `set(Calendar.DAY_OF_MONTH, 1)`

**教训**：
1. `Calendar.set(YEAR/MONTH)` 不会自动重置 `DAY_OF_MONTH`，必须显式设置所有关心的字段
2. 写涉及日期范围查询的代码时，应检查 `Calendar.getInstance()` 后是否完整设置了年/月/日三个维度
3. 同类方法（`getMonthlySummary` vs `getYearlySummary`）之间的写法差异应引起警觉——一个正确一个错误本身就是信号
