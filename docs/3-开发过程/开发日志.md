# MyPurse 开发日志

> **用途**：记录每次开发会话的实际操作——做了什么、改了什么文件、为什么这样改。
> **面向读者**：后续接手的新开发者、新 AI Agent、未来的自己。
> **原则**：宁可多记，不可漏记。脱离对话上下文后也能看懂。按日期 + 会话编号追溯，不设文档版本号。

---

## 关于会话编号

每条日志开头的 `会话 #N` 是全局递增序号，唯一作用：**跨文档交叉引用**。
问题记录中写"修复于会话 #35"，在本文档中搜索 `#35` 即可定位到完整上下文。
编号不随文档版本变化，是稳定的引用锚点。

- `color` 使用 `Long` 类型（非 Compose `Color`），保持 domain 层零 Android 框架依赖

- 内置分类颜色：支出侧暖色系（橙/蓝/紫/红/棕/青/粉/靛蓝/灰），收入侧冷色系（绿/浅绿/青/紫/蓝/黄绿/红/灰）

### 2026-06-21 | 会话 #37：分类管理页闪退修复（V1.0-#30 回归）

**作者**：CodeBuddy

**操作摘要**：
1. 分类管理页点击后闪退，根因是 `CategoryItem` 中自己写了 `Color(category.color.toULong())` 而非使用已有的 `categoryColor()` 工具函数
2. 修复：改为 `categoryColor(category.color)`，与统计页/首页保持一致

**修改文件（1 个）**：

| 文件 | 改动 |
|------|------|
| `CategoryManageScreen.kt` | 导入 `categoryColor`、`Color(category.color.toULong())` → `categoryColor(category.color)`、变量名改 `catColor` 避免与函数名冲突 |

**根因分析**：
- V1.0-#30 曾记录同一 Bug：`Color(argb.toULong())` 会导致 Compose Color 内部 colorSpace 索引越界崩溃
- 会话 #33 已将 `categoryColor()` 工具函数修复为 `Color(argb)`
- 会话 #36 新增代码时未复用已有工具函数，独立实现了相同的错误转换，导致回归

**教训**：
- 有现成工具函数优先复用，不要重复实现。即使看起来"只是简单类型转换"，也可能踩已知坑。

**文档更新**：

### 2026-06-20 | 会话 #22：启动页与图标最终修正 — Compose 层方案 + 自适应图标

**操作摘要**：用户反复反馈启动页和桌面图标与提供的素材不一致。经多轮排查，最终废弃 SplashScreen API 方案，改用 Compose 层启动页 + 自适应图标规范。

**背景**：
- 会话 #20/#21 中使用 SplashScreen API 的 `windowSplashScreenBackground` 设置启动页背景图，但该 API 不支持 bitmap drawable，导致系统 fallback 显示应用图标
- `androidx.core.splashscreen` 库自动拦截启动过程，使 Compose 层渲染延迟
- 桌面图标因未提供 `mipmap-anydpi-v26` 自适应图标配置，Pixel 等原生系统强制套用圆形遮罩，出现"绿色外圈+白色内圈"

**修改文件（9个项目文件 + 2个文档文件）**：

| 文件 | 变更内容 |
|---|---|
| `app/build.gradle.kts` | 移除 `core-splashscreen` 依赖（该库拦截启动过程且 `windowSplashScreenBackground` 不支持 bitmap） |
| `app/src/main/AndroidManifest.xml` | Activity theme 改为 `Theme.MyPurse.Splash` |
| `app/src/main/res/values/themes.xml` | Splash 主题：`windowBackground = #E3F4EB`（与加载界面图顶部一致）；主主题：移除 `windowIsTranslucent` |
| `app/src/main/res/values/colors.xml` | 新增 `ic_launcher_background = #FCF4E2`（与 logopng 四角米黄色一致） |
| `app/src/main/res/mipmap-anydpi-v26/ic_launcher.xml` | **新增** — 自适应图标配置：背景层纯色 `@color/ic_launcher_background`，前景层 `@mipmap/ic_launcher_foreground` |
| `app/src/main/res/mipmap-anydpi-v26/ic_launcher_round.xml` | **新增** — 同上 |
| `app/src/main/res/mipmap-{density}/ic_launcher_foreground.webp` | **新增** — 从 logopng.png 提取前景层（去除米黄背景，保留绿色圆环+白色圆形+账本图案，其余透明），各密度 108/162/216/324/432dp |
| `app/src/main/res/mipmap-{density}/ic_launcher.webp`、`ic_launcher_round.webp` | 从 logopng.png 重新生成（48/72/96/144/192dp） |
| `app/src/main/res/drawable-nodpi/splash_full.png` | 从 `assets/加载界面.png` 复制（MD5 一致） |
| `app/src/main/java/com/wyd/mypurse/MainActivity.kt` | 新增 `AppEntry()` composable：先用 `Image` 渲染 splash_full.png（800ms），再切换 MyPurseNavHost；`onCreate` 开头 `setTheme(Theme_MyPurse)` 切回主主题 |
| `docs/3-开发过程/问题记录.md` | 问题 #20 更新为最终版本（自适应图标+Compose 层启动页） |
| `docs/3-开发过程/开发日志.md` | 新增本次会话记录 |

**关键设计决策**：
- **废弃 SplashScreen API**：该 API 的 `windowSplashScreenBackground` 只支持纯色，`animatedIcon` 会被强制裁剪为圆形。改用 Compose 层直接渲染全屏图，完整保留渐变和计算器图案
- **自适应图标**：从 logopng.png 提取前景层（去米黄背景），配合纯色背景层。Pixel 等原生系统拿到规范的自适应图标后不再额外叠加遮罩
- **冷启动过渡**：系统 preview window 使用 `#E3F4EB` 背景色（与加载界面图顶部一致），视觉过渡不突兀。图标闪现是 Android 系统行为，无法完全消除

**教训**：
- SplashScreen API 不适合展示复杂整页设计图，只适合"纯色背景+小 logo"场景
- 自适应图标必须提供 `mipmap-anydpi-v26/ic_launcher.xml`，否则 Pixel 等原生系统会强制处理
- 用户提供的素材应优先使用原图，避免 AI 二次加工
- 国产手机（华为/小米/OPPO）的自适应图标行为与 Pixel 一致，都遵循 Android 官方规范
