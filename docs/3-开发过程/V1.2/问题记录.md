### V1.2-#1：`getTransactionById()` 仍使用快照名

- **发现日期**：2026-06-23
- **严重程度**：🟡 体验问题
- **所属模块**：数据层 / 交易详情
- **修复**：🔲 未修复

**现象**：
编辑旧交易时，如果分类名已被修改，详情页初始显示的是旧快照名（而非当前分类名）。

**根因分析**：
`getTransactionById()` 是 suspend 函数（非 Flow），无法使用 `combine` 反查实时名。当前调用 `.toDomain()`（无参数版本）直接使用快照名。

**解决方案**：
改为先查 `categoryNameById` 的当前值（`categoryDefDao.getAllCategories().first()`），再传入 `toDomain(nameMap)`。

**涉及文件**：
- `app/.../repository/TransactionRepositoryImpl.kt`：`getTransactionById()`

**教训**：
Flow-based 的 `toDomain(nameMap)` 和 suspend-based 的 `toDomain()` 两套映射，需要逐一排查所有 suspend 查询入口是否也需要实时名。应在实现时就做全量检查。

### V1.2-#2：SQLite GROUP BY 非严格模式下的潜在不确定性

- **发现日期**：2026-06-23
- **严重程度**：🟢 优化建议
- **所属模块**：数据层
- **修复**：🔲 未修复（低风险，暂不处理）

**现象**：
聚合查询的 `GROUP BY t.category_l1_id` 中，SELECT 的 `COALESCE(cd.name, t.category_l1)` 部分依赖快照名 `t.category_l1`。如果同一 `category_l1_id` 下的记录有不同的快照名（如直接修改过数据库），SQLite 会从组中任意取一行，结果不确定。

**根因分析**：
SQLite 默认允许 SELECT 非 GROUP BY 列（`t.category_l1` 不在 GROUP BY 中），但取值是任意的。仅在正常使用（同一分类 ID 的记录快照名一致）时无影响。

**解决方案**：
可考虑 `MAX(t.category_l1)` 或 `MIN(t.category_l1)` 明确聚合语义。当前风险极低（需要直接篡改数据库才会触发），暂不处理。

**涉及文件**：
- `app/.../dao/TransactionDao.kt`：`getTopCategoriesByMonth`、`getCategoryComposition`、`getSubCategoryComposition`

**教训**：
当 GROUP BY 用 ID 但 SELECT 中包含快照字段时，需考虑 SQLite 宽松模式下的不确定性。未来迁到 PostgreSQL 等严格 SQL 引擎时可能直接报错。
