### V1.2-#1：分类改名后历史流水仍显示旧名（快照未同步）

- **发现日期**：2026-06-23
- **严重程度**：🟠 功能缺陷
- **所属模块**：数据层 / 全部展示页面
- **修复**：✅ 已修复（2026-06-23，V1.2 会话 #6，方案B：改名/合并时同步更新快照列）

**现象**：
- 修改分类名后，流水列表显示新名（通过 Flow combine 反查），但点击编辑进入详情页仍显示旧快照名
- CSV 导出也使用旧快照名
- 快照列与 `category_def` 不一致，随时可能出问题

**根因分析**：
`transaction` 表同时存 `category_l1_id`（ID 引用）和 `category_l1`（快照文本）。读路径有两套：Flow 版通过 `combine(nameMap)` 反查实时名，suspend 版直接取快照。写路径（改名）只更新了 `category_def`，没同步更新流水表的快照列。

**解决方案**：
采用方案B（改名时同步更新快照）替代方案A（读时 JOIN 反查）：
1. `CategoryRepositoryImpl.updateCategory()`：改完 `category_def` 后调用 `updateCategoryL1Snapshot/L2Snapshot` 批量更新流水快照
2. `CategoryRepositoryImpl.mergeCategoryAndDelete()`：每次 `updateCategoryL1Id/L2Id` 后同步更新快照
3. `TransactionRepositoryImpl`：删除 `categoryNameById` Flow、`combine` 逻辑、`toDomain(nameMap)` 重载，所有读路径回归最简 `toDomain()` 取快照列

**涉及文件**：
- `app/.../dao/TransactionDao.kt`：新增 `updateCategoryL1Snapshot`、`updateCategoryL2Snapshot`
- `app/.../repository/CategoryRepositoryImpl.kt`：`updateCategory` + `mergeCategoryAndDelete` 增加快照同步
- `app/.../repository/TransactionRepositoryImpl.kt`：删除 nameMap/combine 逻辑，回归 `toDomain()`

**教训**：
1. 快照列的语义应该是"可靠副本"而非"历史存档"。如果写路径不维护它，它就成了陷阱。
2. 两套映射（Flow + suspend）是一个信号，说明数据一致性有问题。应该从源头解决而非到处打补丁。
3. 写操作（改名）频率远低于读操作，把代价从高频路径（显示/导出）移到低频路径（改名）是正确的权衡。

### V1.2-#2：SQLite GROUP BY 非严格模式下的潜在不确定性

- **发现日期**：2026-06-23
- **严重程度**：🟢 优化建议
- **修复**：✅ 已修复（2026-06-23，V1.2 会话 #6，作为 #1 方案B 的附带改进）

**现象**：
聚合查询用 `LEFT JOIN ... COALESCE(cd.name, t.category_l1)` 时，`t.category_l1` 不在 GROUP BY 中，SQLite 松散模式下取值不确定。

**根因分析**：
`LEFT JOIN` + `COALESCE` 的设计意图是兜底孤儿流水（分类已删除但流水仍在）。但当前业务流程保证：删分类必级联删/迁移流水，不存在孤儿数据。

**解决方案**：
伴随 V1.2-#1 方案B 的实施，聚合查询从 `LEFT JOIN ... COALESCE(cd.name, t.category_l1)` 改为 `JOIN ... cd.name`（内连接，直接取实时名），消除不确定性。

**涉及文件**：
- `app/.../dao/TransactionDao.kt`：`getTopCategoriesByMonth`、`getCategoryComposition`、`getSubCategoryComposition`

### V1.2-#3：年视图部分月份不显示 + 分类模式切回时间模式闪退

- **发现日期**：2026-06-23
- **严重程度**：🟠 功能缺陷
- **所属模块**：流水列表
- **修复**：✅ 已修复（2026-06-23，V1.2 会话 #6）

**现象**：
1. 流水列表切换到年视图，只显示最近几个月份（如 4、5、6 月），1-3 月有数据但不显示
2. 先从统计页点击二级分类跳转到流水列表（分类模式），再切换为"时间"分组，App 闪退

**根因分析**：
问题1 由两个 bug 叠加：
- `loadYearView()` 中 `hasMore = transactions.size == pageSize`，但初始加载的 `limit` 是 `maxLoadSize`（500），比较变量用错
- `setGroupMode()` 切换到 TIME 模式时未清除 `categoryL1Id`，导致数据被统计页跳转时传入的旧分类 ID 过滤

问题2 是因为：分类模式下 `categoryL1Id` 保持非 null，切换到 TIME 模式后数据仍按该分类过滤，结合年视图的 `hasMore` 比较 bug 导致状态不一致，引发崩溃。

**解决方案**：
- `setGroupMode()` 中：切换到 TIME 时清除 `categoryL1Id = null`
- `loadYearView()` 中：`hasMore` 改为 `transactions.size == limit`（使用实际 limit 值）

**涉及文件**：
- `app/.../transactionlist/TransactionListViewModel.kt`：`setGroupMode()` + `loadYearView()`

**教训**：
1. 导航参数（如 `categoryL1Id`）应该在切换模式/重置状态时显式清理，不能依赖隐式覆盖
2. `hasMore` 的判断变量应该和实际分页参数一致，用变量而非硬编码常量更安全

### V1.2-#4：移除分页后 `hasMore` 残留导致编译错误

- **发现日期**：2026-06-23
- **严重程度**：🟡 体验问题（非线上 Bug，纯编译期）
- **所属模块**：流水列表
- **修复**：✅ 已修复（2026-06-23，V1.2 会话 #7，批量移除 6 处 `hasMore` 残留）

**现象**：
移除分页逻辑时，`TransactionListUiState.hasMore` 字段已删除，但 ViewModel 中 `loadCategoryView` / `loadYearView` / `loadMonthView` / `loadWeekView` / `loadDayView` / 搜索方法共 6 处 `it.copy(hasMore = false)` 未同步移除，导致编译错误 `No parameter with name 'hasMore' found`。

**根因分析**：
批量替换时遗漏了部分 `it.copy()` 调用中的 `hasMore` 参数。`loadCategoryView` 的 `it.copy()` 块（3 行）与其余方法的 `it.copy()` 块（4 行，含废弃注释）格式不一致，`old_str` 匹配失效导致 1 处漏改；另 4 处因首次修复未再次确认残留导致。

**解决方案**：
逐处搜索并移除所有 `hasMore = false` 残留，共修复 6 处（首轮漏 1 处 + 次轮补 5 处）。

**涉及文件**：
- `app/.../transactionlist/TransactionListViewModel.kt`：6 处 `hasMore = false` → 移除

**教训**：
1. 删除 UiState 字段后，必须全文件搜索该字段名确认所有引用已清理，不能仅凭 lint 通过即认为安全（lint 可能因缓存延迟）
2. `it.copy()` 块中有废弃注释（如 `// 年视图一次性加载，不分页`）导致块结构不一致时，批量 `replace_in_file` 容易漏改
