### V1.3-#1：统计页展开一级分类后，未选二级的流水不显示

- **发现日期**：2026-07-01
- **严重程度**：🟠 功能缺陷
- **所属模块**：统计页
- **修复**：✅ 已修复（2026-07-01，V1.3 会话 #2，新增未分类流水查询并在展开列表中追加"未分类"行）

**现象**：
统计页点击一级分类展开查看二级分类构成时，只显示选了二级分类的流水。未选二级的流水（`category_l2 IS NULL`）在展开列表中完全不可见，导致二级金额之和 < 父级总金额，用户困惑。

**根因分析**：
`getSubCategoryComposition` SQL 使用 `JOIN category_def cd ON t.category_l2_id = cd.id`，INNER JOIN 天然排除 `category_l2_id IS NULL` 的记录。

**解决方案**：
新增 `getUncategorizedSubAmount()` DAO 查询（`SUM(amount) WHERE category_l2 IS NULL`），在 `loadSubComposition()` 中拼入一条 `categoryL1 = "未分类"`、`categoryL1Id = -2L` 的条目。该行在 UI 中继承父级颜色，排在所有二级分类之后。

**涉及文件**：
- `TransactionDao.kt`：新增 `getUncategorizedSubAmount()` SQL 查询
- `TransactionRepository.kt`：新增接口方法
- `TransactionRepositoryImpl.kt`：实现
- `GetStatisticsUseCase.kt`：`loadSubComposition()` 合并未分类金额

**教训**：
涉及 JOIN 的统计查询中，需要考虑 NULL 值被排除的情况。使用 COALESCE 或 UNION + LEFT JOIN 是常见处理方式，但本次场景（只需要一条汇总行）用单独查询更简单，避免了 UNION 对已有查询的侵入。

### V1.3-#2：统计页二级展开列表中进度条与百分比数字不一致

- **发现日期**：2026-07-01
- **严重程度**：🟡 体验问题
- **所属模块**：统计页
- **修复**：🔲 未修复（计划随 V1.4 统计重构一并处理）

**现象**：
展开一级分类查看二级构成时，进度条宽度与右侧百分比数字不匹配。截图中「未分类」¥7,147.08（91.7%）的进度条明显短于文字所示比例；部分小比例项（如 3.1%）的进度条反而显得过长。

**根因分析**：
1. `subPct`（进度条宽度）有 `coerceAtLeast(0.02f)` 强制最小 2% 宽度，当真实占比小于 2% 时条会比数字更宽——但这只能解释小比例项，不能解释 91.7% 的条过短。
2. 更可能的原因：`parentTotal` 取值依赖 `items`（Top 5 一级列表），如果展开时 `items` 与 `subComposition` 的数据状态不同步（如 `items` 用的是旧缓存而 `subComposition` 已更新），`parentTotal` 就可能不一致，导致所有条的比例都错。

**解决方案**：
计划随 V1.4 统计页重构解决：取消「点击展开二级」的交互，改为点击一级分类进入单分类详情页，在详情页中展示二级构成。重构时统一使用同一份数据源的 `total` 作为分母，消除状态不同步风险。

**涉及文件**：
- `ui/statistics/StatisticsScreen.kt`：二级展开渲染逻辑
- `ui/statistics/StatisticsViewModel.kt`：`subComposition` 与 `items` 的并行加载可能导致数据不同步

**教训**：
UI 中两份计算（进度条 vs 文字）应确保使用同一变量作为分母，避免 `parentTotal` 在不同位置重复获取。并行 `async` 加载后应确保数据一致性。

### V1.3-#3：Smart cast 失败——自定义 getter 属性无法被 Kotlin 智能转换

- **发现日期**：2026-07-01
- **严重程度**：🟡 编译错误（非运行时）
- **所属模块**：记账页
- **修复**：✅ 已修复（2026-07-01，V1.3 会话 #3，改用 Elvis 操作符 `(amountDecimal ?: BigDecimal.ZERO) > BigDecimal.ZERO` 绕开 smart cast）

**现象**：
```kotlin
enabled = uiState.amountDecimal != null && uiState.amountDecimal > BigDecimal.ZERO
```
编译报错：`Smart cast to 'java.math.BigDecimal' is impossible, because 'amountDecimal' is a property that has an open or custom getter.`

**根因分析**：
`amountDecimal` 在 `AddTransactionUiState` 定义中使用了自定义 getter：
```kotlin
val amountDecimal: BigDecimal?
    get() = amount.toBigDecimalOrNull()
```
Kotlin 编译器的 smart cast 只能用于无自定义 getter 的 `val` 属性。原因：自定义 getter 每次调用都可能返回不同值——第一次调用 `!= null` 返回非 null，第二次调用 `.>` 时 getter 又被调用，编译器无法保证两次返回同一对象引用（虽然实际逻辑上不会变，但编译器只做静态分析）。同理，`val` 属性如果定义在另一个模块或类中（非 `final`），也会被编译器拒绝 smart cast。

**解决方案**：
改用 Elvis 操作符，避免对同一属性做两次求值：
```kotlin
enabled = (uiState.amountDecimal ?: BigDecimal.ZERO) > BigDecimal.ZERO
```
`?:` 会将左侧求值结果直接用于右侧比较，不涉及 smart cast。

**解决方案变体**（任选其一）：
1. 赋值给局部变量：`val amount = uiState.amountDecimal; enabled = amount != null && amount > BigDecimal.ZERO`
2. Elvis：`(uiState.amountDecimal ?: BigDecimal.ZERO) > BigDecimal.ZERO`
3. 将 getter 改为带 backing field 的初始化赋值（如果逻辑允许）

**涉及文件**：
- `ui/addtransaction/AddTransactionScreen.kt`：退款按钮 enabled 条件

**教训**：
1. 在 Compose UI 或 ViewModel 中访问 data class 中带有自定义 getter 的属性时，不要指望 Kotlin 能 smart cast。应提前将值存到局部变量，或使用 `?.` / `?:` 等安全操作符链式处理。举一反三：任何非直接 `val` 声明的属性（包括 `val ... get() = ...`、open val、接口属性），都可能触发此问题。
2. **Agent 自省**：写 `!= null &&` 模式时必须回看目标属性的声明方式。本次疏忽不是经验问题，是增量代码的"二次确认"环节没做到位——只关注了逻辑正确性，没有反向追溯属性定义确认编译器能否 smart cast。改进：凡是涉及属性判空后直接使用的代码，先确认属性是否为纯 `val` 声明。
