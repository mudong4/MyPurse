## 0. 设计前置原则

> 从 V1.0 开发教训中提炼。任何涉及**跨页面共用元素**的改动，编码前必须先执行全局影响分析。

**触发条件**（满足任一即触发）：
- 涉及颜色值、色板、颜色映射
- 涉及时间/日期选择器
- 涉及分类数据展示（图标、颜色、名称格式）
- 涉及多个页面共用的 UI 组件

**执行流程**：
1. 列出所有使用该元素的页面/组件
2. 分析当前各自如何实现（是否重复、是否一致）
3. 确定统一方案（抽取公共组件 or 统一数据源 or 统一色板变量）
4. 方案确定后再开始编码

**V1.0 典型反例**：
- 日历/时间选择器：记一笔、流水列表、统计页各实现一套（至少 3 套），交互不一致、代码重复、修改需逐个改
- 分类颜色：5 个页面各自决定颜色逻辑，无统一数据源
- 颜色硬编码：`TransactionDetailScreen.kt` 有 5 处硬编码颜色，`AppColors.kt` 中已有对应变量但未使用

## 2. V1.0.1 体验优化（规划中）

> **执行状态追踪**：逐项进度（✅/🔲）以 `docs/3-开发过程/开发阶段.md` 阶段 10 为准。本章仅描述需求方向和设计意图。

### 2.1 分类管理交互优化

**现状问题**：
1. 分类列表展示"眼花缭乱"，不够直观
2. 只有点击分类文字才能展开二级，点击箭头或空白区域无反应
3. 二级分类无自定义排序能力

**优化方向**（2026-06-21 讨论确定）：
- 展开/折叠的点击热区扩大到整行（含拖拽手柄区域）
- 二级分类用 ▲▼ 上下移动按钮替代拖拽（二级项数少 3-8 个，按钮比拖拽更合适，且避免与一级长按拖拽手势冲突）
- 视觉层次优化：一级分类左侧加颜色竖条（利用 `Category.color`），二级区域加背景区分，父子归属更清晰

### 2.2 统计页时间选择器统一

**现状问题**：
- 统计页点击时间标签弹出的是菜单式选择（日/周/月/季/年），选"年"和"月"是分别弹独立对话框
- 流水列表页的时间选择是滚轮式（WheelPicker），交互更直观
- 两处不一致，且统计页的年/月分开选择体验差

**优化方向**：
- 统计页时间选择器改为与流水列表一致的滚轮式（YearMonthWheelSheet）
- 年/月合并为一个滚轮面板，而非分开两个对话框
- 评估是否可抽取公共组件（当前两处使用，按规范 2.3.1 的 ≥2 即抽取规则，应统一为公共组件；但需评估参数设计避免膨胀）

### 2.3 UI 风格优化

**现状问题**：
- 当前 UI 风格有优化空间，希望做成"简约又不失美观"
- 参考 `ui-style-demo` 中的风格

**优化方向**：
- V1.0.1：先做一套默认的优化风格（全局配色、圆角、间距、阴影等视觉打磨）
- V1.1：实现**视觉主题体系**，在设置页统一切换。包括：
  - 明/暗模式（跟随系统或手动切换）
  - 多套预设配色方案（清新绿、暖橙、深邃蓝、极简灰等）
  - 用户自定义分类颜色和图表配色

> **说明**：V1.0 原需求中"暗黑模式"和"主题风格"是分开列的两项，实际它们同属视觉主题体系。暗黑模式是明/暗维度的切换，配色方案是在此基础上叠加的色彩变量，统一在设置页的"主题风格"入口下管理。

### 2.4 数据可视化增强

**现状问题**：
- 本月支出排行 TOP5 颜色单一
- 统计页构成分析中不同分类颜色区分不够
- 缺少百分比和支出金额的直观展示
- 条形图和饼图展示可优化
- 趋势图有优化空间

**优化方向**：
- 为不同分类分配固定颜色（餐饮=橙、交通=蓝、购物=紫等），首页排行和统计页统一
- 条形图/饼图：显示金额 + 占比%，颜色与分类对应
- 趋势图：评估是否增加预算参考线、动画过渡等
- 颜色体系需预留扩展性，为 V1.1 用户自定义颜色做准备

## 3. V1.1 多套主题风格 + 固定收支自动记账

> **V1.1 定位**：提供 5 套预设主题风格（每套自适应系统深色模式） + 用户自定义分类/图表颜色 + 固定收支自动记账。核心交付物是统一的视觉主题引擎和可靠的自动记账系统。
> **执行状态**：本章内容为需求描述。开发进度以 `开发阶段.md` 阶段 11 为准。

---

### 3.1 多套预设主题风格

**暗黑模式策略**：不提供 App 内手动明暗切换。每套预设自带浅色和深色两套配色方案，深色模式跟随手机系统设置自动切换。用户只需选择喜欢的预设风格，明暗维度由系统处理。

V1.1 将当前已有的主题基础设施（`ThemePreset`、`ChartColorScheme`）升级为完整的、用户可在设置页切换的多风格主题引擎。

**配色范围说明**：多套预设只改变 Material3 框架色（primary/secondary/background/surface 等）和图表色（chartPalette）。**AppColors 功能语义色（AppExpenseRed、AppIncomeGreen 等）不随主题预设变化**——支出红/收入绿是通用语义约定，在不同预设间保持稳定，避免用户认知混乱。

**当前基础设施**（V1.0.1 已就绪）：
- `ThemePreset` 数据类：将 lightScheme + darkScheme + chartColors 打包为一个可切换单元
- `DefaultPreset`、`SoftPreset` 两套预设。**注意**：`SoftPreset` 为 V1.0.1 阶段的占位预设（色值复用 `LightColorScheme`），V1.1 引入 5 套正式预设后，`DefaultPreset` 吸纳为「默认（紫）」、`SoftPreset` 退役（无独立色值，本身即占位符）
- `LocalThemePreset`：主题预设偏好（默认 DefaultPreset）
- `category_def.color` 列：分类专属颜色（V1.0-#39 修复后已回填默认值）

#### 3.1.1 预设清单与选择

| 预设名 | 风格定位 | 主色调 | 适用偏好 |
|--------|---------|--------|---------|
| 🟢 清新绿 | 自然、轻盈 | 绿 + 灰白 | 偏好清爽风格 |
| 🔵 深邃蓝 | 沉稳、专注 | 蓝 + 深灰 | 偏好冷静风格 |
| 🟠 暖橙 | 温暖、亲切 | 橙 + 米白 | 偏好柔和风格 |
| ⚫ 极简灰 | 克制、中性 | 灰 + 白 | 偏好无干扰 |
| 🟣 默认（紫） | 当前配色 | 紫 + 白 | 新用户默认 |

> 每个预设需要定义独立的一套 `lightColorScheme` + `darkColorScheme` + `ChartColorScheme`。命名色值（primary/secondary/tertiary/background/surface 等）应体现该风格的性格，而非仅换主色。

**交互设计**：

| 项目 | 描述 |
|------|------|
| 入口 | 设置页「显示」分组下，「主题风格」选项行，显示当前预设名 + 小型颜色点预览 |
| 选择器 | 点击弹出 BottomSheet，展示预设列表（预设名 + 颜色预览环 + 当前选中标记） |
| 存储 | DataStore 持久化预设名（String 类型，key: `theme_preset_name`） |
| 生效 | 通过 `LocalThemePreset` 向下传递，`MyPurseTheme` 读取 |
| 默认值 | 新用户默认 = "默认（紫）" |

**技术约束**：
- 预设的 lightScheme/darkScheme 必须以 Material3 `lightColorScheme()`/`darkColorScheme()` 构建，确保所有语义 slot 正确填充
- chartColors 的 palette/paletteExtended 数组长度必须 ≥ 6 且 ≥ 10，兜底色（最后的灰色）必须保留

#### 3.1.2 用户自定义分类颜色

**功能描述**：用户可修改任意分类（内置或自定义）的专属颜色，覆盖默认色和主题预设的图表轮转色。

**交互设计**：

| 项目 | 描述 |
|------|------|
| 入口 | 分类管理页 → 长按一级分类 / 点击二级分类 → 操作菜单中选择「修改颜色」 |
| 颜色选择器 | 弹出 BottomSheet，展示预设色盘（12-16 个推荐色块） + 「自定义」按钮 |
| 自定义取色 | 点击「自定义」→ 弹出 HSV 色相环取色器（Compose 原生实现或引入轻量库），实时预览 |
| 预览 | 颜色选择器中显示该分类名 + 当前颜色预览，修改时即时变化 |
| 存储 | 更新 `category_def.color` 字段（ARGB Long 值） |
| 重置 | 内置分类提供「恢复默认」按钮，将 color 还原为 `CategoryDefaults` 中定义值；自定义分类无默认色，不显示此按钮 |
| 新建 | 新建分类时，在名称输入后增加颜色选择步骤（可选，默认取当前主题 chartPalette 的可用色） |

**全局生效**：
- 所有展示分类颜色的页面（记一笔分类选择器、流水列表、首页 TOP5、统计页构成分析、分类管理页）统一从 `category.color` 取值
- `chartPalette` 的轮转 fallback 逻辑保留，用于 color=0 的极端情况兜底

#### 3.1.3 用户自定义图表配色

**功能描述**：用户可自定义统计图表的配色序列（非分类专属色，而是图表整体的色调风格）。

**交互设计**：

| 项目 | 描述 |
|------|------|
| 入口 | 设置页「显示」分组下，「图表配色」选项行 |
| 选项 | ① 跟随主题（默认）、② 自定义序列 |
| 自定义 | 提供类似分类颜色选择器的色盘，用户可逐个修改 palette 中的 6-10 个颜色 |
| 存储 | DataStore 持久化自定义 palette（JSON 编码的颜色值列表） |
| 生效范围 | 统计页构成分析环形图/条形图、趋势图柱状/折线、首页 TOP5 排行 — 所有使用 `chartPalette` 的地方 |

### 3.2 固定收支自动记账

> 需求来源：`requirements.md` 第 3.7 节（V1.0 推迟）。以下为基于原始需求的细化，补充 V1.1 场景下的交互和数据层设计。

#### 3.2.1 功能概述

用户可创建固定收支模板（周期 + 金额 + 分类），App 每次启动时自动检查并生成对应流水记录。覆盖房租、订阅、工资等周期性固定收支。

| 场景 | 示例 | 周期 |
|------|------|------|
| 每月固定支出 | 房租、房贷、物业费、订阅会员 | 每月 X 日 |
| 每周固定支出 | 每周五聚餐、每周一交通充值 | 每周 X（周一至周日） |
| 每年固定支出 | 保险年费、域名续费、会员年费 | 每年 X 月 X 日 |
| 每日固定支出 | 每日早餐预算、通勤费 | 每天 |

#### 3.2.2 数据模型

`recurring_template` 表（已规划于 `requirements.md` 第 5.4 节）：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | Long (PK, auto) | 模板 ID |
| flowType | String | 支出 / 收入 |
| categoryL1Id | Long? | 一级分类 ID |
| categoryL2Id | Long? | 二级分类 ID（可选） |
| amount | BigDecimal | 金额 |
| note | String? | 备注 |
| cycleType | String | DAILY / WEEKLY / MONTHLY / YEARLY |
| cycleValue | Long | 周期参数（日=几号，周=周几 1-7，月=几号，年=月*100+日）。SQLite 列名 `cycle_value`，Entity 属性名 `cycleValue` |
| isActive | Boolean | 是否启用。SQLite 列名 `is_active`，Entity 属性名 `isActive` |
| lastExecutedDate | Long? | 上次执行日期时间戳 |
| nextExecutionDate | Long | 下次执行日期时间戳（预计算） |
| createTime | Long | 创建时间 |

#### 3.2.3 触发与执行

| 环节 | 描述 |
|------|------|
| 触发时机 | 每次 App 冷启动时执行一次检查，由独立 `RecurringScheduler` 类负责。不依赖后台服务 |
| 执行逻辑 | ① 查询所有 `isActive=true` 的模板；② 对每个模板，计算 `nextExecutionDate ≤ 今天` 的所有应执行日期；③ 对每个应执行日期，检查是否已有同一 `recurringTemplateId` 的记录（防重复）；④ 无则创建一条 TransactionRecord，`recurringTemplateId` 设为模板 ID；⑤ 更新 `lastExecutedDate` 和 `nextExecutionDate` |
| 补记策略 | 若错过多个周期（用户多日未打开），一次性补记所有遗漏周期。最大补记范围：过去 90 天 |
| 无效日期处理 | 如「每月 31 日」，当月无 31 日时自动退到当月最后一天 |
| 性能约束 | 自动记账检查不增加启动耗时超过 200ms（`requirements.md` 6 章性能要求） |

#### 3.2.4 模板管理页（新页面 G）

| 项目 | 描述 |
|------|------|
| 入口 | 设置页「数据管理」分组下 → 「固定收支模板」 |
| 列表展示 | 所有模板列表，每条显示：周期标签（每月/每周/…）、分类名、金额、启用/禁用开关 |
| 新建模板 | 点击「+」→ 表单页：选择流水类型 → 选择分类 → 输入金额 → 选择周期类型 → 设置周期参数（如每月几号）→ 可选备注 → 保存 |
| 编辑模板 | 点击模板 → 进入编辑页（同新建表单，预填充已有值） |
| 删除模板 | 长按或滑动删除，确认弹窗：「删除此模板，已生成的记录不受影响。确定删除？」 |
| 启用/禁用 | 列表行右侧 Switch，即时切换 `isActive` |
| 空状态 | 无模板时显示「暂无固定收支模板」+ 新建按钮 |

#### 3.2.5 与现有功能的联动

| 位置 | 行为 |
|------|------|
| 流水列表 | 自动生成的记录显示 🔄 标记（`recurringTemplateId != null`） |
| 流水详情/编辑 | 编辑自动记录时，顶部提示「此记录由固定模板「xxx」自动生成。编辑后将不再自动更新。」保存后 `recurringTemplateId` 置为 null |
| 流水删除 | 删除自动记录时，弹窗提示「此记录由固定模板自动生成，删除后模板仍会在下次到期时生成新记录。」（与手动记录删除弹窗文案不同） |
| 首页概览 | 自动生成的记录在支出/收入汇总中正常计入，无特殊标记 |
| 统计页 | 自动生成的记录正常参与统计，无特殊标记 |

### 3.3 推迟项（V1.2+ 或更远期）

> 以下功能原列于 V1.0 规划中，V1.1 聚焦视觉体系 + 自动记账，暂不纳入。

| 功能 | 推迟理由 | 建议版本 |
|------|---------|---------|
| 桌面小部件 | Widget 框架独立于 App 主流程，开发成本高 | V1.2+ |
| 按一级分类设预算 | 需扩展预算表结构（当前仅单条），且统计页需对应改造 | V1.2 |
| 自定义流水类型 | 流水类型从二值扩展为 N 种，需改造记一笔页 UI（从 Tab 切换改为下拉/滚轮）、分类管理页、统计页筛选逻辑 | V1.2 |
| 首页自定义布局 | 交互设计复杂度高（拖拽排列、卡片显隐），且 V1.0 首页已可用 | V1.3+ |

### 3.4 分类管理页多展示方式（远期设想）

> 2026-06-21 讨论记录。当前默认使用 Card + 展开/折叠（手风琴）模式，已足够覆盖 8-9 个一级分类的场景。以下为远期可选方向，待用户反馈后决定优先级。

| 展示模式 | 适用场景 | 说明 |
|---|---|---|
| Card 展开/折叠（当前） | 一级分类 ≤15 个，偶尔管理 | 简洁、空间高效，展开后直接看二级 |
| 两级独立页面（主-从导航） | 一级分类多，需专注管理 | 点击一级 → 跳转新页展示二级，层次清晰但路径长 |
| 分组吸顶列表 | 需一目了然看到全部父子关系 | 一级作为 sticky header，二级为普通行，信息密度高 |
| 纯网格模式 | 快速浏览，不常编辑 | 类似记一笔页的分类选择器，所有一级铺开展示 |

**远期目标**：在分类管理页顶部增加展示方式切换入口（如 SegmentedButton），用户可根据使用习惯和分类数量自行选择。各模式共用同一套编辑/删除/排序操作，仅布局不同。

## 附录 A：全局影响分析基线（2026-06-21）

> 本次分析覆盖三个维度：颜色体系、时间选择器、分类展示。分析结果作为 V1.0.1 开发的参考基线，后续各子阶段编码前应回顾此基线。

### A.1 颜色体系

**当前架构**：项目已有三层颜色架构（`Color.kt` → `AppColors.kt` → `Theme.kt`），整体设计良好。大部分页面正确使用语义变量。

**硬编码问题**：`TransactionDetailScreen.kt` 有 5 处硬编码颜色，`AppColors.kt` 中均有对应变量：

| 硬编码值 | 应替换为 | 出现次数 |
|---|---|---|
| `Color(0xFFE53935)` | `AppExpenseRed` | 2 |
| `Color(0xFF43A047)` | `AppIncomeGreen` | 1 |
| `Color(0xFFFFF3E0)` | `AppEditWarningBg` | 1 |
| `Color(0xFFE65100)` | `AppEditWarningText` | 1 |

**分类颜色各自为政**：5 个页面对分类颜色的处理各自独立，无统一数据源：

| 页面 | 颜色来源 | 特点 |
|---|---|---|
| 首页 TOP5 排行 | `AppBudgetBlue` 固定蓝色 | 所有分类同色 |
| 统计页构成分析 | `chartPalette[index]` 按排名轮询 | 颜色与分类名无关 |
| 记一笔分类选择 | `surfaceVariant` 统一灰色 | 无区分 |
| 分类管理页 | `surfaceVariant` 统一灰色 | 无区分 |
| 流水列表 | 支出红/收入绿 二元 | 仅区分收支方向 |

**统一方向**：在 `Category` 数据模型中增加 `color` 字段，各页面从数据源取色，而非各自决定。

### A.2 时间选择器

**当前状态**：至少 3 套独立实现，4 种年份选择方式，交互不一致：

| 组件 | 位置 | 行数 | 风格 |
|---|---|---|---|
| `ChineseDatePickerDialog` | `ui/components/`（公共） | ~280 | Dialog |
| `CalendarSheet` | `StatisticsScreen.kt`（私有） | ~220 | BottomSheet |
| `YearMonthPickerDialog` | `TransactionListScreen.kt`（私有） | ~80 | AlertDialog |
| `YearPickerDialog` | `TransactionListScreen.kt`（私有） | ~50 | AlertDialog |
| `YearSheet` | `StatisticsScreen.kt`（私有） | ~50 | BottomSheet |
| `QuarterSheet` | `StatisticsScreen.kt`（私有） | ~80 | BottomSheet |

**统一方向**：
- V1.0.1：统计页时间选择器改为与流水列表一致的滚轮式（`YearMonthWheelSheet`），年月合并
- 长期：评估是否抽取公共组件（当前仅两处使用，暂不强抽）

### A.3 分类展示

**当前状态**：`Category` 模型无 `color`/`icon` 字段。流水列表用分类名首字符充当"图标"，各页面自行决定展示逻辑。

**统一方向**：
- V1.0.1：`Category` 增加 `color` 字段，建立分类→颜色映射
- V1.1：`Category` 增加 `icon` 字段，支持用户自定义

### A.4 分析结论

V1.0.1 开发建议顺序：
1. **先修硬编码**：`TransactionDetailScreen.kt` 5 处 → 替换为 `AppColors` 变量（改动小、无风险）
2. **先定颜色体系**：`Category` 加 `color` 字段 → 各页面统一取色 → 再做排行/统计图可视化
3. **时间选择器统一**：独立改造统计页，不改颜色时不动它
4. **分类管理页交互**：等具体方案确定后再编码
